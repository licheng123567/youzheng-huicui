【HIGH优先级】需立即检查 — 涉及数据安全边界

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #1 菜单隐藏 = API权限校验（导航层隐藏→功能层强制）                          ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：前端菜单隐藏了"成员管理"，但直链 /settings/members 仍可访问        ┃
┃                                                                           ┃
┃ 检查：                                                                    ┃
┃   □ 后端所有路由是否有 @Permission/@Perm 装饰器？                         ┃
┃   □ 尝试以受限角色直链访问，应被拒访或重定向                             ┃
┃   □ API调用是否二次校验权限（不仅仅依赖前端 v-if）？                    ┃
┃                                                                           ┃
┃ 易错：导出/下载操作仅前端隐藏，后端无权限校验                            ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #2 只读数据行无写操作按钮（脱敏数据◐ → 按钮隐藏）                         ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：结案案件对VL只读可见(◐)，但"标注/标记/转质检"按钮仍显示            ┃
┃                                                                           ┃
┃ 检查：                                                                    ┃
┃   □ 表格行遍历时，按钮是否叠加行权限检查？ v-if="can() && canEdit()"     ┃
┃   □ 脱敏行上的批量勾选是否被禁用？                                        ┃
┃   □ 质检看板中只读提示(◐)的行是否隐藏了"处置"按钮？                    ┃
┃                                                                           ┃
┃ 易错：前端隐藏可见，但后端API未校验（点击仍能调用）                       ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #3 查询语句含租户过滤（WHERE org_id=? 强制过滤）                           ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：同一查询对不同角色应返回不同数据集，但缺少租户范围过滤               ┃
┃                                                                           ┃
┃ 检查：                                                                    ┃
┃   □ 后端 DAO/Service 所有 SELECT 语句是否包含 WHERE org_id=?              ┃
┃   □ SE查询时是否过滤数据范围？ WHERE property_id IN (SELECT ...)         ┃
┃   □ 统计/聚合查询是否在 GROUP BY 前加范围过滤？                          ┃
┃   □ 跨表 JOIN 时是否逐表验证范围？                                       ┃
┃                                                                           ┃
┃ 易错：前端过滤正确，后端API忽略参数 | 子表未加范围限制 | 统计无过滤        ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #4 高危操作二次确认+受控原因（撤案/删除/结案时弹Modal+枚举）              ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：撤案、删除、结案等操作无二次确认或原因非受控枚举（自由文本）         ┃
┃                                                                           ┃
┃ 检查高危操作清单：撤案、坏账结案、冲正回款、作废案件、批次关闭              ┃
┃ 每个操作需验证：                                                          ┃
┃   □ 是否弹出二次确认Modal？不仅仅是 console.warn()                       ┃
┃   □ 原因字段是 <select options={CFG.REASONS}> 还是 <input type="text">  ┃
┃   □ 禁用提交直到选择原因？ :disabled="!selectedReason"                  ┃
┃   □ 后端是否校验原因值？validate(reason IN CFG.REASONS)                 ┃
┃                                                                           ┃
┃ 易错：直接执行未弹窗 | 原因为自由文本 | 对话仅前端，API无验证              ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #5 批量操作逐项权限检验（批量派单→for-loop逐项校验，非一次权限检查）      ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：后端批量API仅校验全局权限一次，未逐项验证每条数据的所有权/范围       ┃
┃                                                                           ┃
┃ 检查：                                                                    ┃
┃   □ POST /batch-dispatch 内是否 for (caseId in caseIds) { 逐项校验 } ？   ┃
┃   □ 逐项校验包含：权限 + 数据范围 + 状态？                                ┃
┃   □ targetVL 是否被校验为"授权的服务商"？                                ┃
┃   □ 前端勾选框是否仅允许勾选"有权操作的"行？:disabled="!canEdit()"       ┃
┃                                                                           ┃
┃ 易错：未校验targetVL | 未逐项检查范围 | 前端勾选无限制                    ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #11 结案脱敏规则全覆盖（结案后VL/CO重定向到统计视图，禁进详情+明细）      ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：案件结案后VL/CO仍能访问原详情页，看到完整通话/承诺/工单明细         ┃
┃                                                                           ┃
┃ 检查流程（选一个已结案案件）：                                             ┃
┃   □ 以VL/CO身份点击案件，是否自动跳转到"脱敏统计视图"而非详情页？         ┃
┃   □ 统计视图是否仅显示计数/转化率，无完整明细？                           ┃
┃   □ 若强制进详情页URL，是否被拒访？                                      ┃
┃   □ 列表中业主联系方式是否脱敏(****)?                                   ┃
┃   □ 代码：if (case.isClosed && (CO|VL)) return <StatView> 非 <DetailView> ┃
┃                                                                           ┃
┃ 易错：详情页可访问 | 子标签页仅隐藏未完全脱敏 | 直链可绕过                 ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #12 用户故事全链路走查（不同角色完整操作一次关键流程）                    ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 场景：催收员(CO)结案全流程                                               ┃
┃   1. 登录→工作台(应见本商待办)  □ PASS                                   ┃
┃   2. 进案件→详情(应见通话/承诺)  □ PASS                                  ┃
┃   3. 标注→登记承诺(权限检查)  □ PASS                                     ┃
┃   4. 结案(二次确认+原因下拉)  □ PASS                                     ┃
┃   5. 返回列表(案件状态变更)  □ PASS                                      ┃
┃   6. 再进案件(自动转脱敏视图)  □ PASS ← 关键检查点                      ┃
┃   7. 查操作日志(仅见本案/本商)  □ PASS                                   ┃
┃                                                                           ┃
┃ 任何环节失败则整体审计失败。建议每个大版本走一遍。                        ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛


【MEDIUM优先级】需本周内检查 — 涉及操作追溯与跨端一致

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #6 代理操作审计日志有痕迹（SA代PL结案时，日志须记proxyFor字段）           ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：SA代理PL操作时，审计日志未记录"谁被代操作了"，无法追溯               ┃
┃                                                                           ┃
┃ 检查：                                                                    ┃
┃   □ SA选择角色切换器→代理PL-A→结案案件                                  ┃
┃   □ 查看该操作日志，是否包含 proxyFor/onBehalfOf/targetOrg 字段？        ┃
┃   □ 日志格式例：{ operator: "SA-Alice", proxyFor: "PL-Bob", op: "结案" }  ┃
┃   □ 代码检查：logOp() 函数是否接收 proxyFor 参数？                       ┃
┃                                                                           ┃
┃ 易错：日志无proxyFor | 权限缓存未刷新导致仍用SA权限 | localStorage未清除   ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #7 跨端权限口径一致（PC/App/H5权限规则同步或文档明确差异）               ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：同一操作在PC网页、App、H5的权限规则不一致                           ┃
┃                                                                           ┃
┃ 检查关键操作（结案/标注/派单）：                                           ┃
┃   □ CO在App标注→PC网页脱敏规则是否一致？                                ┃
┃   □ H5业主页token是否在账单支付/撤案后失效？                             ┃
┃   □ PL在App的菜单与PC是否相同？                                         ┃
┃   □ 差异是否有文档说明（为何App无"成员管理"）？                          ┃
┃                                                                           ┃
┃ 易错：App缺少dataRange参数 | H5 token失效机制缺失 | 菜单差异无文档         ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛

┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ #8 权限点字典与代码绑定完整（权限定义↔后端装饰器↔前端检查，无孤岛）       ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：权限字典定义了权限，但前端/后端代码中的检查不一致导致孤岛/混淆      ┃
┃                                                                           ┃
┃ 检查步骤：                                                                ┃
┃   1. 列出所有权限点：case.dispatch, case.mark, case.promise, ...          ┃
┃   2. 对每个权限点验证三点：                                               ┃
┃      □ 后端有 @Permission('case.xxx') 装饰器？                           ┃
┃      □ 前端有 can('case.xxx') 或 v-if 检查？                            ┃
┃      □ 该权限是否真的被某个功能用到（非孤岛）？                          ┃
┃   3. 反向检查：grep 代码中所有 can() 调用，确认都在权限字典中定义          ┃
┃                                                                           ┃
┃ 易错：定义case.close但代码检查case.result | 权限名大小写不一致              ┃

┃ #10 API参数目标对象校验（派单targetVL/转移targetProperty要校验授权）      ┃
┣━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┫
┃ 问题：API请求虽过权限验证，但参数中的目标对象可能用户无权操作               ┃
┃                                                                           ┃
┃ 检查涉及"指定目标"的API：                                                 ┃
┃   □ POST /case/dispatch { caseIds: [...], targetVL: 'xxx' }              ┃
┃      验证：targetVL 是否属于 user.authorizedVLs？                        ┃
┃   □ POST /case/transfer { caseId, targetProperty }                       ┃
┃      验证：targetProperty 是否属于 user.orgId？                          ┃
┃   □ POST /batch/assign { employeeId }                                    ┃
┃      验证：employeeId 是否属于本商？                                     ┃
┃                                                                           ┃
┃ 易错：仅校验"caller有权限"，未校验"目标对象被授权"                        ┃

快速诊断树

权限问题？
  ├─ "看得见但操作不了" → #1, #2, #3 (导航/数据隔离)
  ├─ "按钮灰显而非隐藏" → #2 (应隐藏非禁用)
  ├─ "跨商户看到别人数据" → #3 (租户过滤缺失)
  └─ "高危操作没提示" → #4 (二次确认缺失)

脱敏问题？
  ├─ "结案后仍见详情" → #11 (脱敏重定向缺失)
  ├─ "列表中联系方式未脱敏" → #11 (字段脱敏缺失)
  └─ "跨端脱敏规则不一致" → #7 (跨端同步缺失)

操作追溯问题？
  ├─ "不知谁代操作的" → #6 (代理日志缺失proxyFor)
  └─ "无法追溯操作原因" → #4 (原因非受控枚举)

跨租户泄漏？
  ├─ 快速排查 #3 (SQL WHERE子句) + #12 (用户故事走查)
