## 🎯 核心12条检查项

### 🔴 HIGH优先级（7项）— 必须通过

```
1️⃣  菜单隐藏 = API权限校验         → grep @Perm + 直链测试
2️⃣  只读数据无写操作按钮           → v-if检查 + 按钮点击测试
3️⃣  查询语句含租户过滤             → grep WHERE org_id
4️⃣  高危操作二次确认+受控原因       → Modal+枚举验证
5️⃣  批量操作逐项权限检验           → for-loop逐项校验
11️⃣ 结案脱敏规则全覆盖            → 结案后重定向验证
12️⃣ 用户故事全链路走查            → 完整业务流程走查
```

### 🟠 MEDIUM优先级（4项）— 本周内

```
6️⃣  代理操作审计日志有痕迹         → proxyFor字段检查
7️⃣  跨端权限口径一致              → PC/App/H5对齐
8️⃣  权限点字典与代码完整          → 权限定义↔代码映射
10️⃣ API参数目标对象校验           → 目标授权检查
```

### 🟡 LOW优先级（1项）— 优化

```
9️⃣  演示数据覆盖多租户            → mock数据租户数量检查
```

## 📊 典型审计结果

### 优秀(一级，12/12)
```
✓ 所有菜单隐藏都有后端权限校验
✓ 脱敏数据行无操作按钮
✓ 所有SELECT都有WHERE org_id
✓ 所有高危操作都弹二次确认
✓ 批量操作都逐项校验权限
→ 可直接上线
```

### 良好(二级，11/12 或 10/12 with MEDIUM失败)
```
✗ 某个权限点缺@Perm装饰器 (HIGH)
✓ 其他11项通过
→ 修复后可上线
```

### 需改(三级，<10/12 HIGH通过)
```
✗ 权限缺陷2-3个
✓ 数据隔离基本通过
→ 必须修复，修复工作量 1-3天
```

### 禁交(一级缺陷，<7/12通过)
```
✗ 多个HIGH项失败(跨租户泄漏/权限越权)
→ 禁交付，需重新设计权限架构
```

## 🛠️ 快速修复参考

| 缺陷 | 修复代码行 | 预计时间 |
|---|---|---|
| 缺@Permission装饰器 | `@Permission('perm.key')` | 5min |
| SQL缺WHERE org_id | `WHERE org_id=#{orgId}` | 5min |
| 脱敏重定向缺失 | `if(closed) router.push('/stats')` | 10min |
| 高危操作无确认 | `Modal.confirm()` + 枚举 | 10min |
| 批量无逐项校验 | `for(item) { validate(item) }` | 15min |
| 代理无proxyFor | `logOp(..., proxyFor)` | 5min |

## ❓ 常见问题

**Q: 这个体系适用于我的项目吗？**
A: 适用于**所有多租户SaaS系统**。12条检查项是通用的，具体代码示例按你的技术栈调整。

**Q: 审计需要专业的安全团队吗？**
A: 不需要。任何开发/测试人员按《执行工单模板》逐项执行即可。前提是理解权限模型。

**Q: 发现严重缺陷该怎么办？**
A: 参考《快速修复工作表》，大多数缺陷5-15分钟内可修复。

**Q: 我如何衡量审计成效？**
A: 看总分。0-6/12严重；7-10/12可接受；11-12/12优秀。

**Q: 权限矩阵和审计体系什么关系？**
A: 权限矩阵=规范(应该怎样)；审计体系=验证(实际是否如此)。两者缺一不可。
