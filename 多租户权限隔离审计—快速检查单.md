# 多租户SaaS 分角色权限隔离审计检查单

> 为指导"角色一致性审计"，提供8-12条**可直接操作的核对项**。每条包含问题描述、检查路径、常见遗漏点。适用于代码审查、测试、上线前审计阶段。

## 详细检查项说明

### 1. 菜单隐藏 = API权限校验 ✓

**问题**：前端导航隐藏了某菜单（如"成员管理"），但直链访问 `/settings/members` 或调用 API 仍可越权操作。

**检查路径**：
- [ ] 列举所有导航菜单项，记录隐藏条件（如 `role !== 'SA'`）
- [ ] 尝试以受限角色直链访问对应页面，应被拒访或重定向
- [ ] 检查后端路由是否有权限装饰器（如 `@Permission('members.view')`）
- [ ] 遍历后端源码，grep `@Perm` 或 `@Permission`，确认每个路由都有守卫

**常见遗漏**：
- 导出/下载操作前端隐藏，但API无权限点校验
- 直链参数篡改后仍可访问（如 `?org=xxx` 被绕过）

---

### 2. 只读数据行无写操作按钮 ✓

**问题**：某行数据对用户只读可见（◐脱敏），但"编辑/删除/转移"按钮仍显示，导致权限越界。

**检查路径**：
- [ ] 在列表/表格中，找出所有应为只读的行（如结案案件、其他物业批次）
- [ ] 点击该行查看，验证"编辑/删除/转移/标注"类按钮是否已隐藏
- [ ] 代码检查：数据行遍历内的按钮需叠加权限条件
  ```js
  // ✗ 不安全：仅检查全局权限
  <button v-if="can('case.mark')">标注</button>
  
  // ✓ 安全：检查行权限 + 全局权限
  <button v-if="can('case.mark') && canEditThisRow(item)">标注</button>
  ```

**常见遗漏**：
- 表格的批量勾选按钮无权限限定（可勾选无权限的行）
- 质检看板中只读提示(◐)的行仍有"转质检"按钮

---

### 3. 查询语句含租户过滤 ✓

**问题**：不同角色查询同一对象时，应返回不同数据集合。但代码缺少 `WHERE org_id=?` 之类的范围约束，导致跨租户泄漏。

**检查路径**：
- [ ] 打开后端 DAO/Service 层源码
- [ ] 每条 SELECT/UPDATE/DELETE 语句前，检查 WHERE 子句是否包含租户字段
  ```sql
  -- ✗ 不安全
  SELECT * FROM cases WHERE status='进行中'
  
  -- ✓ 安全
  SELECT * FROM cases WHERE status='进行中' AND property_id=${propertyId}
  ```
- [ ] SE(平台员工)查询应包含：`WHERE property_id IN (SELECT ... FROM dataRange WHERE userId=${currentUserId})`
- [ ] 批量查询(如报表汇总)应在聚合函数前过滤租户

**常见遗漏**：
- 统计接口（回款率/在催数）无数据范围过滤
- 对账单查询前端过滤正确，但后端API忽略了参数
- 跨表 JOIN 时仅在主表加范围，子表未加

---

### 4. 高危操作二次确认+受控原因 ✓

**问题**：撤案、删除、结案等危险操作应弹出确认对话，且"原因"应从预定义枚举中选择（非自由文本），以便审计追溯。

**检查路径**：
- [ ] 列举所有高危操作：撤案、坏账结案、冲正回款、作废案件、批次关闭等
- [ ] 对每个操作：
  - 是否弹出 Modal 二次确认？ `v-if="showConfirmDialog"`
  - 原因字段是否为受控枚举？ `<select options={CFG.CLOSE_REASONS}>`
  - 是否禁用"提交"直到用户选择了原因？ `:disabled="!selectedReason"`
- [ ] 后端是否校验了操作权限和原因有效性？

**常见遗漏**：
- "作废"直接执行，未弹确认
- 结案时虽然弹了确认，但原因是 `<input type="text">` 而非下拉
- 冲正回款未标注"操作不可逆"

---

### 5. 批量操作逐项权限检验 ✓

**问题**：用户批量勾选 N 条数据做操作（如批量派单），后端仅校验了一次权限(caller 拥有 permission.key)，但未逐项验证每条数据的所有权和范围。

**检查路径**：
- [ ] 找到所有批量操作 API（如 POST /case/batch-dispatch）
- [ ] 检查后端实现：
  ```js
  // ✗ 不安全
  if (user.can('case.dispatch')) {
    for (const caseId of caseIds) {
      dispatch(caseId); // 未校验每项数据所有权
    }
  }
  
  // ✓ 安全
  for (const caseId of caseIds) {
    const caseData = getCase(caseId);
    validate(caseData.property IN user.dataRange); // 检查范围
    validate(caseData.status === '待派单');        // 检查状态
    validate(user.can('case.dispatch'));           // 检查权限
    dispatch(caseId);
  }
  ```
- [ ] 前端是否只允许勾选"当前用户有权操作的"行？（禁用 disabled）

**常见遗漏**：
- 批量派单时未校验 targetVL 的服务商授权
- 批量标记风险时仅校验了全局权限，未验证每项数据所有权
- 批量下载时对 SE 无数据范围过滤

---

### 6. 代理操作审计日志有痕迹 ✓

**问题**：SA代PL执行操作时（如SA代结案），操作日志应明确记录"谁被代操作了"，否则无法追溯。

**检查路径**：
- [ ] 找到"代理模式"的触发点（通常是角色切换器）
- [ ] 执行一次代理操作（如SA选择代理PL-A后结案某案件）
- [ ] 查看该操作的审计日志，应包含以下字段：
  ```json
  {
    "operator": "SA-Alice",      // 执行者
    "operation": "案件结案",
    "targetCaseId": 12345,
    "proxyFor": "PL-Bob",         // ← 被代人必须有！
    "proxyOrgId": "物业A",        // ← 被代组织必须有！
    "timestamp": "2026-06-19 10:23:45"
  }
  ```
- [ ] 代码检查：`logOp()` 或类似审计函数是否接收 proxyFor 参数？

**常见遗漏**：
- 审计日志只记了操作，没有 proxyFor 字段
- SA 代操作时权限缓存未刷新，导致查询逻辑仍用 SA 的范围而非被代人的范围
- 前端 localStorage 缓存了旧角色，切换后未清除

---

### 7. 跨端权限口径一致 ✓

**问题**：PC Web、Mobile App、H5 业主页中，同一权限操作的规则应保持一致。差异应文档明确。

**检查路径**：
- [ ] 列出所有"跨端"操作（如CO打电话、标注、结案）
- [ ] 对每个操作，在三端分别执行，观察：
  - 权限门控是否一致（能/不能做）
  - 字段显隐是否一致
  - 脱敏规则是否一致
- [ ] 例：CO 在 App 结案后，是否与 PC 网页中的脱敏规则相同？
- [ ] H5 业主页使用 token 而非登录，token 失效机制是否完善？

**常见遗漏**：
- App 端 CO 的工作台与 PC 私海数据范围不一致
- H5 的 token 在用户改手机后仍有效(应失效)
- PC 端菜单隐藏了，但 App 仍可点击

---

### 8. 权限点字典与代码绑定完整 ✓

**问题**：后端定义了权限点（如 `case.dispatch`），但前端/后端的实际代码中检查的权限点可能不一致，导致孤岛或混淆。

**检查路径**：
- [ ] 找到权限点字典定义（通常在 `backend/config/permissions.ts` 或数据库表）
- [ ] 列出所有权限点，例如：
  - `case.dispatch`（派单）
  - `case.mark`（标注）
  - `case.promise`（承诺）
  - `case.evidence`（存证）
  - ...
- [ ] 对每个权限点，检查：
  - 后端是否有对应的 `@Permission('case.xxx')` 装饰器？
  - 前端是否有对应的 `can('case.xxx')` 或 `v-if="hasPerm(...)"` ？
  - 是否存在"定义了但从未被检查"的孤岛权限？
  - 是否存在"代码中检查了，但权限字典中未定义"的权限？
- [ ] 每个角色是否仅拥有"最少必需权限"？grep role_permissions 表

**常见遗漏**：
- 权限字典中有 `case.close`，但代码中检查的是 `case.result`（名字不一致）
- SE 被无意中授予了 `case.evidence`（存证）权限（应无）
- 某些高危操作（如冲正回款）无权限点，直接检查了 role

---

### 9. 演示数据覆盖多租户 ✓

**问题**：系统使用 mock 数据做演示时，若所有数据都属于同一租户，权限隔离的差异会在演示中"隐形"，无法验证跨租户过滤。

**检查路径**：
- [ ] 查看演示用的 mock 数据（通常在 `index.html L2500+` 或 `mock.json`）
- [ ] 验证是否包含至少 3 个物业/服务商/项目
- [ ] 以不同角色登录，检查数据过滤是否真实生效
  - SE 应仅见授权范围内的数据
  - PL-A 应仅见物业A的数据
  - VL-B 应仅见服务商B的数据
- [ ] 如果演示中因数据同租户导致权限差异不可见，应在审计报告中标注"演示弱，逻辑真实存在"

**常见遗漏**：
- mock 数据只有"阳光花园"物业，无法验证"物业隔离"
- 所有服务商都是同一家，无法验证"跨商隐藏"

---

### 10. API参数目标对象校验 ✓

**问题**：某些 API 请求虽然权限通过了认证，但请求参数本身可能包含用户无权操作的目标（如派单指定了不属于授权的服务商）。

**检查路径**：
- [ ] 找出所有涉及"指定目标对象"的 API（如派单、转移、分配）
  - `POST /case/dispatch { caseIds: [...], targetVL: 'xxx' }`
  - `POST /case/transfer { caseId: xxx, targetProperty: 'yyy' }`
- [ ] 检查后端是否校验了参数中的目标对象：
  ```js
  // ✗ 不安全
  if (user.can('case.dispatch')) {
    dispatch(targetVL); // 未验证 targetVL 是否被授权
  }
  
  // ✓ 安全
  if (user.can('case.dispatch')) {
    validate(targetVL IN user.authorizedVLs); // 验证服务商授权
    validate(targetVL.status === 'active');   // 验证目标状态
    dispatch(targetVL);
  }
  ```

**常见遗漏**：
- 派单时后端未校验 targetVL 对应服务商是否被授权
- 转移案件时未验证 targetProperty 是否属于 PL
- 分配催收员时未验证目标员工属于本商

---

### 11. 结案脱敏规则全覆盖 ✓

**问题**：案件进入"已结清/撤案/坏账"状态后，VL/CO 对该案件的访问权限应大幅收窄：不能看详情、不能看子数据，仅能看脱敏统计视图。

**检查路径**：
- [ ] 选择一个已结案的案件
- [ ] 以 VL/CO 身份尝试访问该案件的详情页：
  - [ ] 是否被重定向到"脱敏视图"而非标准详情页？
  - [ ] 原详情页的所有标签页（通话记录、承诺、工单等）是否隐藏？
  - [ ] 是否显示"结案统计视图"（仅计数、转化率、成本等统计值）？
- [ ] 在列表中查看该案件的行：
  - [ ] 业主联系方式列是否显示 `****`(脱敏)？
  - [ ] 其他敏感字段（减免金额、律师函）是否脱敏？
- [ ] 代码检查：结案后的渲染逻辑
  ```js
  // ✓ 结案后 VL/CO 重定向到统计视图
  if (case.isClosed && (role === 'VL' || role === 'CO')) {
    return <CaseStatsView case={case} />; // 不是详情页
  }
  ```

**常见遗漏**：
- 结案后详情页仍可访问（直链未被阻止）
- 列表中的敏感字段未脱敏
- 子数据(通话记录/承诺/工单)虽然隐藏了字段，但仍显示为 `—` 而非完全不渲染

---

### 12. 用户故事全链路走查 ✓

**问题**：前述各项审计可能出现"局部正确、整体错误"。需要以真实用户身份完整走一次关键业务流程，观察每个环节的权限/脱敏表现。

**检查路径**：

**案例：催收员(CO)结案全流程**
1. [ ] **登录** → 进工作台，观察"今日待标注"任务列表是否仅显示"本商公海+私海"的案件
2. [ ] **打开案件** → 列表→案件详情页面，观察：
   - 能否看到"通话/转写"标签？（应能）
   - 能否看到"承诺/工单"标签？（应能）
   - 能否看到"操作日志"？（应仅见与本案相关的，非跨商）
3. [ ] **标注跟进** → 点"标注结果"，能否填写？（应能，权限检查 `can('case.mark')`）
4. [ ] **登记承诺** → 点"登记承诺"，弹出表单，能否提交？（应能）
5. [ ] **结案** → 点"结案"按钮
   - [ ] 弹出确认对话？（必须）
   - [ ] 原因从下拉选择？（必须，非自由文本）
   - [ ] 二次确认后案件状态变为"已结清"？
6. [ ] **返回列表** → 刷新，观察该案件是否从列表消失或标记为"已结清"
7. [ ] **再进案件** → 点击已结清案件，观察：
   - [ ] 是否自动重定向到"脱敏统计视图"而非详情页？
   - [ ] 若能进详情页，是否隐藏了"通话/承诺/工单"等明细标签页？
8. [ ] **查看日志** → 列表→操作日志，验证"结案"记录是否可见

**其他关键流程**：
- **平台员工派单**：SE 应仅见授权范围内的批次，派单给不授权的服务商时应被拒
- **物业对账**：PL 进对账页面，应仅见本物业的对账单，无法看到"付佣对账"
- **角色切换**：SA 选择代理 PL，切换后权限应立即生效，尝试派单应被限制在该 PL 的范围

**常见遗漏**：
- 流程中某个环节权限缓存未刷新，导致跨租户泄漏
- 脱敏在列表有效，但详情页未刷新
- 代理操作的审计日志无痕迹

## 关键要点总结

**三道防线**：
1. **导航与权限一致** ← 检查项 #1：菜单隐藏是否后端验证
2. **数据可见与写操作隔离** ← 检查项 #2, #3, #11：只读数据、租户过滤、结案脱敏
3. **高危操作的受控流程** ← 检查项 #4, #6：二次确认、代理追溯
