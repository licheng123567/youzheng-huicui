# 有证慧催 PRD · M1 组织与权限（RBAC）

> 依赖：[00-总纲](./00-总纲.md) §4 角色总览、§9 多租户隔离。
> 目标：定义三方组织结构、账号与登录、角色、功能授权与数据可见范围。
> **契约对齐**：本模块与 API 契约 v1.0.0 对齐。

---

## 1. 模块范围

- 三方组织（平台/物业/服务商）的建立与归属。
- **账号与登录模型**（账号标识、密码/验证码登录、手机号换绑、一号多账号）。
- 角色层级：**单一负责人 + 员工**（无中间主管层）。
- 平台按用户体系给员工**功能授权**（菜单/操作级）。
- **数据可见范围（多租户隔离）**的硬规则。
- 账号生命周期（启用、停用、重置）、**负责人交接（改绑手机+重置密码）**。

**不在本模块**：具体业务操作权限点的全集（散落在各业务模块，本模块给授权机制与隔离规则）。

---

## 2. 关键概念

| 概念 | 说明 |
|---|---|
| 组织（租户） | 一个物业公司或一个服务商，是数据隔离边界；平台是超级租户。**数据归组织所有、组织内共享，不按个人隔离** |
| 负责人 | 物业/服务商组织**唯一**的最高权限账号，可管理本组织成员与授权；**换人靠改绑手机+重置密码交接同一账号**（不设账号转移）。平台侧对应"平台超管" |
| 员工 | 负责人/超管创建、被授予功能子集的账号（物业=协调员、服务商=催收员、平台=平台员工） |
| 账号 | 每人一个独立账号，归属唯一组织；**登录标识 = 账号名（唯一、稳定）** |
| 权限点 | 可被授予的最小功能单元（菜单可见/操作可执行） |
| 角色模板 | 一组权限点的预设集合（如"协调员""催收员""平台员工"），便于批量授权 |

## 3. 业务规则

### 3.1 组织与负责人
- **BR-M1-01**：组织三类——平台（唯一，超级租户）、物业（多个）、服务商（多个）。物业与服务商均由**平台**创建后开通（创建时不录签约信息，佣金比例后续按批次维护）。
- **BR-M1-02**：每个物业/服务商组织**有且仅有一个负责人账号**（唯一）；平台侧最高权限为平台超管。**负责人换人通过给该账号改绑手机号 + 重置密码交接（同一账号），不设"账号转移到另一账号"机制**；平台超管或现任负责人可操作，数据归属不变。
- **BR-M1-02a（数据归组织）**：组织内项目/批次/案件/充值余额等数据**归组织所有、组织内有权成员共享**；成员离职/更换**不影响数据归属与可见性**。
- **BR-M1-03**：平台提供**预设角色模板**（协调员/催收员/平台员工）作为起点；**负责人可在其自身权限范围内给员工分配功能/权限**（员工权限是负责人权限的**子集**，可在范围内增减）；员工不可自行扩权。
- **BR-M1-03a（物业协调员 = 执行层权限边界）**：物业协调员默认侧重**执行**（案件/送达/工单/法务申请/线下回款标记/缴费链接/跟进登记），**默认不含以下"管理/财务/审计"职能，归物业负责人**：①**批次/案件导入**（M2 BR-M2-14）；②**计费明细**（能力消耗：STT/短信/存证，M9）；③**操作日志/审计**（§9）；④**收（付）佣对账的确认/驳回**（M9，协调员对对账**仅只读查看**）；⑤**项目档案/收费标准/作战手册正式版维护**（M2 BR-M2-11）。负责人可在其权限范围内按需给协调员增减（BR-M1-03）。
- **BR-M1-04**：平台超管可配置/启用角色模板，并可对任意组织成员授权（用于支持与兜底）。**注：成员账号的创建/停用以 BR-M1-04a 为准——平台只建管平台员工、对跨组织成员仅只读；此处"代物业/代服务商管理"仅指代操作业务动作（撤案/充值等 BR-M1-15），不含代建/代停成员账号。**
- **BR-M1-04a（成员账号创建归属）**：平台**只创建/管理平台员工**；服务商、物业的协调员/催收员由其**本组织负责人自建**，平台对跨组织成员仅**只读查看**（不代建、不停用，违规处置归各组织负责人 M5 BR-M5-07a）。组织新建时只设唯一负责人账号，员工由负责人在组织内自建。
- **BR-M1-04b（平台参数配置范围）**：平台超管「参数配置」含 **CFG 时效/计费参数 + 角色模板 + 权限矩阵总览 + AI 能力配置中心（模型密钥 / 通话前后提示词 / 作战手册飞轮后台，详见 M5 BR-M5-13）**；平台员工(SE)默认不授权、需超管显式授予（§8）。
  - **BR-M1-04c（权限矩阵总览）**：参数配置提供「**权限矩阵总览**」——功能点 × 6 角色（●可操作 / ◐只读 / ○不可见 / ▲需授权），平台可查看、**可导出**，作为授权边界的可视化总览（细粒度授予仍在成员管理按子集勾选）。
  - **BR-M1-04d（未授权 UI 口径统一）**：SE 等**未授权**功能的界面行为统一为「**入口可见但只读/禁用 + "需平台授权"提示**」（**非不渲染**），便于知晓功能存在并申请授权；AI 配置中心 / 话术库 / 参数配置 / 新建组织/员工等均遵此口径。
- **BR-M1-14（平台员工双维度授权）**：平台员工的权限含两个维度——①**功能权限点**（可操作哪些功能）；②**数据范围**（可看/可管哪些对象，可限定为某些**区域** / 指定**服务商** / 指定**物业公司**）。超出数据范围的对象对该员工**不可见**。默认数据范围 = 全平台，超管可收窄；范围内的功能仍受功能权限点约束。
- **BR-M1-15（平台代操作）**：平台超管可代物业/服务商执行操作，**允许范围包含代充值、代发缴费链接、代撤案/坏账结案等**（结案不设审核流）；**每次代操作均生成操作记录**（操作人、代谁、动作、时间），且**被代组织事后可见该记录**（透明性）。结案类代操作另见 M8。

### 3.2 账号与登录
- **BR-M1-09（登录标识）**：登录标识为**账号名**（唯一、稳定，不随人员变动而变）；不以手机号作为唯一标识。
- **BR-M1-10（登录方式）**：支持两种登录——①账号名 + 密码；②手机号 + 短信验证码。
- **BR-M1-11（手机号规则）**：手机号**可随时更换**，且**可重复**——同一手机号可绑定多个账号（如协调员与催收员同号、或一人在多组织/多角色）。因此：
  - 手机验证码登录时，若该手机号绑定多个账号，系统**列出可登录账号供用户选择**。
  - 一个手机号对应多个角色/账号是**允许且正常**的场景。
- **BR-M1-13（跨组织多身份）**：一个自然人可在**不同组织各持一个独立账号**（如：某物业公司的协调员 + 同时兼职某服务商的催收员），属**多账号**而非单账号跨组织。各账号：
  - **归属各自唯一组织、数据严格隔离、互不串数据**（受 BR-M1-05 约束）；
  - 通过**同一手机号关联**，验证码登录时按 BR-M1-11 选择账号；
  - 在哪个账号登录，就只见/只操作该组织数据。
- **BR-M1-13a（多身份对组织负责人不可见 = 后台处理）**：跨组织多身份的**关联与权限隔离由平台后台处理**，**对组织负责人不暴露**——物业/服务商负责人在「成员管理」**只见本组织成员**，**不感知**该成员是否在其他组织兼职（保护成员隐私、避免泄露他组织信息）。故「成员管理」中**新建员工仅在本组织内**、不提供跨组织关联入口；**跨组织兼职账号视图仅平台（SA）可见**。

### 3.3 数据隔离
- **BR-M1-05（数据隔离）**：
  - 物业成员只能看到**本物业**的项目/批次/案件与对账。
  - 服务商成员只能看到**派给本服务商**的案件 + 本服务商私海；**看不到其他服务商**数据。
  - 催收员只能看到**本服务商公海** + **自己私海**；仅"开放抢单"案件可跨服务商可见（M3）。
  - **物业 ↔ 服务商 无任何资金/对账数据互通**（强约束，来自 §2 商业模式）。
- **BR-M1-06**：服务商成员可见的物业侧数据**仅限催收依据**：物业↔业主合同、收款信息、物业费标准、作战手册、减免规则（详见 M2）。不含物业的佣金/对账。**案件结案（结清/撤案/坏账）后，服务商与催收员对该案件收敛为"只看统计、不看明细"**（业主信息/联系方式/跟进/录音等明细不可见，明细仅平台+物业可见，详见 M8 BR-M8-09）。

### 3.4 账号生命周期
- **BR-M1-07**：员工账号停用后立即失去登录与所有数据访问；其持有的私海案件须按 M3 释放规则处理（停用即触发释放回公海）。
- **BR-M1-08**：所有授权变更、账号启停、登录、负责人交接（改绑/重置）均留操作日志（§9 全程留痕）。
- **BR-M1-12（负责人账号常驻）**：负责人账号**不停用/不删除**（组织常在）；换人通过改绑手机号 + 重置密码交接同一账号（BR-M1-02）。

## 4. Epic 与用户故事

### Epic M1-A：组织开通与负责人账号

**US-M1-01 平台开通物业组织**
- 作为**平台超管**，我想创建物业公司组织并设置其负责人账号，以便物业能登录并开始建项目。
- 验收标准：
  - **Given** 我是平台超管，**When** 我填写物业公司名称/联系人并创建负责人账号（**不录签约信息**），**Then** 系统生成该物业组织与唯一负责人账号，负责人可凭初始凭据登录（BR-M1-01/02）。
  - **Given** 物业组织已创建，**When** 负责人首次登录，**Then** 其数据可见范围被限制为本物业（BR-M1-05）。
  - **Given** 物业名称重复，**When** 我提交，**Then** 系统提示冲突且不创建。

**US-M1-02 平台开通服务商组织**
- 作为**平台超管**，我想创建服务商组织并设置其负责人账号，以便服务商能接单并管理催收员。
- 验收标准：
  - **Given** 我是平台超管，**When** 我创建服务商并设负责人账号，**Then** 生成服务商组织与唯一负责人账号（BR-M1-02）。
  - **Given** 服务商负责人登录，**When** 其查看数据，**Then** 仅见派给本服务商的案件与本服务商私海（BR-M1-05）。
  - **Given** 服务商负责人尝试访问任何物业对账或其他服务商数据，**When** 请求发起，**Then** 被拒绝（无入口、无数据）。

### Epic M1-B：账号与登录

**US-M1-08 账号登录（账号名/验证码）**
- 作为**任一用户**，我想用账号名+密码或手机验证码登录，以便在换手机号后仍能稳定登录。
- 验收标准：
  - **Given** 我有账号名与密码，**When** 我用账号名+密码登录，**Then** 登录成功。
  - **Given** 我的手机号绑定了**唯一**账号，**When** 我用手机号+验证码登录，**Then** 直接登入该账号（BR-M1-10）。
  - **Given** 我的手机号绑定了**多个**账号（如协调员与催收员同号），**When** 我用手机号+验证码登录，**Then** 系统列出可登录账号供我选择，选定后登入（BR-M1-11）。
  - **Given** 我更换了绑定手机号，**When** 用新手机号验证码登录，**Then** 登录成功；登录标识（账号名）不变（BR-M1-09/11）。

### Epic M1-C：成员管理与授权

**US-M1-03 负责人创建员工并分配角色/权限**
- 作为**物业负责人/服务商负责人**，我想创建员工、套用预设模板并在我权限范围内分配功能/权限，以便员工按职责使用系统。
- 验收标准：
  - **Given** 我是组织负责人，**When** 我创建员工、**绑定其手机号**、套用预设模板（如"协调员"/"催收员"）并**在我权限范围内增减其功能/权限**，**Then** 员工获得相应权限，且**不超过我自身权限范围**（BR-M1-03）。
  - **Given** 我创建员工，**When** 未填写手机号，**Then** 系统阻止并提示手机号为必填（创建即绑手机号）。
  - **Given** 我填写的手机号在他组织已有账号，**When** 我在本组织创建，**Then** 系统**静默**为本组织创建独立账号、与既有账号数据隔离，**不向我（负责人）暴露该手机号在其他组织的兼职情况**（多身份由平台后台处理，BR-M1-13a）。
  - **Given** 员工已分配角色，**When** 其访问该角色外功能，**Then** 系统拒绝且记录日志。

**US-M1-04 平台员工授权（功能 + 数据范围）**
- 作为**平台超管**，我想给平台员工授予功能权限并限定其数据范围（区域/服务商/物业），以便按片区或客户分工运营。
- 验收标准：
  - **Given** 我是超管，**When** 我给平台员工授予"撮合派单"但不授予"参数配置"，**Then** 该员工可派单但看不到参数配置入口。
  - **Given** 我把某平台员工数据范围限定为"华东区 + 服务商A/B"，**When** 该员工浏览撮合/案件/报表，**Then** 仅见范围内区域与服务商/物业的数据，范围外对象不可见（BR-M1-14）。
  - **Given** 某对象超出员工数据范围，**When** 该员工通过任意入口/直链访问，**Then** 被拒且不泄露其存在。
  - **Given** 平台员工，**When** 其执行被授权操作，**Then** 操作成功并留痕（操作人=该员工）。

**US-M1-05 账号启停与重置**
- 作为**负责人/平台超管**，我想启用/停用/重置员工账号，以便管理人员变动。
- 验收标准：
  - **Given** 某催收员持有 N 件私海案件，**When** 负责人停用其账号，**Then** 账号立即不可登录，且其私海案件按 M3 规则**释放回公海**（BR-M1-07）。
  - **Given** 账号被停用，**When** 该用户尝试登录，**Then** 登录被拒并提示账号停用。
  - **Given** 被停用者是跨组织兼职（在其他组织另有账号），**When** 本组织停用其账号，**Then** 其在**其他组织的账号不受影响**，可正常登录使用（BR-M1-13）。
  - **Given** 被停用催收员名下案件含承诺/待办，**When** 案件随停用释放回公海，**Then** 承诺/待办等催收过程数据**随案保留**，新催收员抢单后可看到完整催收历史（BR-M4-10）。
  - **Given** 负责人重置员工密码，**When** 员工用新凭据登录，**Then** 登录成功。

**US-M1-09 负责人交接（改绑手机/重置密码）**
- 作为**平台超管 / 现任负责人**，我想给负责人账号改绑手机号、重置密码，以便负责人换人时同一账号交接、不丢数据。
- 验收标准：
  - **Given** 负责人换人，**When** 我给负责人账号**改绑新手机号、重置密码**，**Then** 新负责人用同一账号登录，组织数据归属不变（不设账号转移机制）（BR-M1-02、BR-M1-02a）。
  - **Given** 改绑/重置发生，**When** 查看日志，**Then** 记录操作（操作人、时间）（BR-M1-08）。

### Epic M1-D：数据可见范围（隔离校验）

**US-M1-06 跨租户访问拦截**
- 作为**平台**，我想确保任何跨组织数据访问被拦截，以便满足资金/隐私隔离硬约束。
- 验收标准：
  - **Given** 服务商 A 的催收员，**When** 其尝试通过任意入口/链接访问服务商 B 的案件，**Then** 返回无权限且不泄露数据存在性。
  - **Given** 物业用户，**When** 其尝试访问任何佣金对账中"平台↔服务商"数据，**Then** 无此入口（BR-M1-05）。
  - **Given** 服务商用户查看案件，**When** 打开案件详情，**Then** 可见物业↔业主合同/收款信息/物业费标准/作战手册/减免规则（BR-M1-06），但不可见物业佣金数据。

**US-M1-07 平台代管操作留痕**
- 作为**平台超管**，我想在必要时代物业/服务商执行操作（代撤案/坏账结案、代充值、代发缴费链接等），以便兜底支持，且操作可追溯、被代方可见。
- 验收标准：
  - **Given** 我是超管，**When** 我代某物业执行撤案/坏账结案（M8），**Then** 操作成功且记录"平台代操作 + 代谁 + 操作人 + 时间"（BR-M1-15）。
  - **Given** 某服务商分钟数不足，**When** 我代其充值，**Then** 充值生效且生成代操作记录（BR-M1-15）。
  - **Given** 业主需缴费链接，**When** 我代某物业/服务商发送缴费链接，**Then** 链接送达且生成代操作记录（BR-M1-15）。
  - **Given** 平台对某组织做过代操作，**When** 被代组织查看其操作记录，**Then** 可见该条代操作（操作人/动作/时间），即透明可追溯。
