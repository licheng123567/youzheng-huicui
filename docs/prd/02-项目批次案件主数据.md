# 有证慧催 PRD · M2 项目-批次-案件主数据

> 依赖：[00-总纲](./00-总纲.md) §5 核心对象与导航主轴、§9 隔离规则。
> 目标：定义 **项目(小区) → 批次(催收单) → 案件** 三层主数据结构、字段、导航下钻与各角色可见范围。
> **契约对齐**：本模块与 API 契约 v1.0.0 对齐。

---

## 1. 模块范围

- 项目（小区）档案的创建与维护。
- 批次（催收单）的录入/导入与挂载。
- 案件的字段、来源、与批次的从属关系。
- **项目→批次→案件** 的逐层下钻、按批次号直达、筛选与聚合。
- 各角色在项目维度的用法与可见字段。

**不在本模块**：案件的池归属/流转（M3）、催收动作（M4）、结案（M8）。

---

## 2. 三层对象与字段

### 2.1 项目（小区）档案
由**物业负责人**创建与维护，含以下分区：

| 分区 | 内容 | 对服务商可见 |
|---|---|---|
| 基本信息 | 小区名称、地址；**物业公司主体（诉讼原告）：公司名称、统一社会信用代码、法定代表人、联系电话**、管理户数等 | ✅ |
| 区域（**省 / 市 / 区县**，三级；取自小区地址） | 用于平台员工数据范围与报表聚合维度；亦作诉状管辖与坐落要素 | ❌（仅平台用，不对服务商，BR-M2-09） |
| 物业合同（物业↔业主） | 物业服务合同要点（催收法律依据）；**合同类型**（包干制/酬金制/前期物业…）+ **合同附件上传**（诉讼证据材料） | ✅（催收依据，BR-M1-06） |
| 缴费周期 & 物业费标准 | 来自物业服务合同的**文字描述**；**物业费标准可按业态多行**（住宅/商铺/车位等各自标准）+ **收费周期** + **违约金**；仅作催收依据；**非金额计算源**（每户实际应收以批次明细为准） | ✅（催收依据） |
| 收款信息（物业↔业主） | 业主向物业缴费的收款账户/通道信息（**文本描述**：开户行/账号/账户名/微信收款渠道等） | ✅（催收依据） |
| 协调员 | 关联本项目的物业协调员（**可多个**） | ❌ |
| 作战手册 | 催收话术/指引/常见异议处理 | ✅（催收依据） |
| 减免政策 | **阶梯式**减免/分期政策（折扣/减免上限/含违约金减免/决定权，BR-M2-18a） | ✅（催收依据） |

### 2.2 批次（催收单）
物业一次性提交的一组欠费案件，挂在某项目下。

| 字段 | 说明 |
|---|---|
| 批次号 | 全局唯一、**可读编码 = 项目缩写 + 日期 + 流水**，可作对账凭证号、可独立直达（§4） |
| 所属项目 | 必填 |
| 提交时间 | 物业上传时间，T1 计时起点（M3） |
| 案件数 / 应收总额 | 聚合值 |
| 作战手册 / 减免政策 | **继承项目**或**批次级自定义覆盖**；项目更新时已覆盖批次保留自定义+可同步（BR-M2-18b）；执行优先级 批次 > 项目 |
| **佣金比例（平台←物业）** | 该批次的收佣比例，平台↔物业协商、按批次维护（CFG-COMM-IN，详见 M9） |
| **付佣比例（平台→服务商）** | 该批次派给某服务商后的付佣比例，按"服务商×批次"维护（CFG-COMM-OUT，详见 M9） |
| 状态 | 待派单 / 已派单 / 部分结案 / 全部结案（聚合自案件） |

### 2.3 案件
单个欠费户的催收对象。

| 字段 | 说明 |
|---|---|
| 案件号 | 全局唯一 |
| 所属批次 / 项目 | 必填，决定导航归属 |
| 业主/户信息 | **必填：户号、姓名、手机号、房号**；及"要素化起诉"所需被告/欠费要素（见 BR-M2-15） |
| 欠费周期 | **必填**，欠费起止周期（要素化起诉的事实要素） |
| 应收金额 | 原始欠费**来自批次明细（按房屋逐户给定）**；减免/分期后形成"减免后应收"（M8 结清判定基准） |
| 池归属 | 平台公海 / 服务商公海 / 私海（M3，本模块只展示不流转） |
| 案件状态 | 待派单 / 进行中 / 已结清 / 已撤案 / 坏账 / 已作废（仅待派单可作废·误传纠错，BR-M2-17）（M8） |
| 跟进摘要 | 最近跟进记录（M4） |

## 3. 业务规则

- **BR-M2-01**：案件必属于唯一批次，批次必属于唯一项目（严格三层从属）。
- **BR-M2-02**：批次号全局唯一，采用**可读编码规则 = 项目缩写 + 日期 + 流水号**（可作物业/服务商对账凭证号）；支持不经项目直达其案件列表（§4）。编码各段格式 TBD。
- **BR-M2-03**：项目档案中标记为"催收依据"的分区对服务商可见（BR-M1-06）；协调员、对账等不对服务商开放。
- **BR-M2-04**：批次提交即作为 T1 派单时限计时起点（M3 接管流转）。
- **BR-M2-05**：减免/分期会改变案件"减免后应收"，该值是 M8 结清判定与 M7 缴费链接金额的基准。
- **BR-M2-06**：物业可对同一项目挂多个批次；项目/批次层级实时聚合下层指标，含：**案件数、应收总额、回款额、回款率、已结清数、在催数、法务处理数、催收进度**。
- **BR-M2-07**：导航主轴在全平台一致——撮合、公海、报表均支持按项目分组与按批次筛选/下钻。
- **BR-M2-08（批次佣金比例）**：批次承载两项比例——收佣比例（平台←物业，CFG-COMM-IN）、付佣比例（平台→服务商，按"服务商×批次"，CFG-COMM-OUT）；均按批次维护，是 M9 结算基准。比例数值本轮 TBD。
- **BR-M2-09（项目区域属性）**：项目含**区域**属性（**省/市/区**，取自小区地址），作为平台员工数据范围（BR-M1-14）与经营报表的聚合/筛选维度；区域不属于"催收依据"，不对服务商展示。
- **BR-M2-10（案件唯一性）**：同一业主在不同批次/缴费周期重复欠费 = **新案件，仅批次不同**，不跨批次合并/累加（承接 M0 §5）。同一业主可对应分属不同批次的多个案件，各自独立流转、催收、结案。
- **BR-M2-11（项目维护权）**：项目档案的**创建/编辑/停用仅物业负责人**可操作；协调员**不可维护项目档案**（仅按指派处理工单/法务/线下回款标记等）。创建项目时**可同时指定该项目的协调员**，之后也可增减（US-M2-02）。
- **BR-M2-12（应收来源）**：案件原始应收金额**来自批次明细（逐户给定）**，**不由"物业费标准"自动计算**；"缴费周期&物业费标准"仅为催收依据的文字描述。
- **BR-M2-13（协调员关联，多对多）**：协调员可关联到**项目级**与**批次级**，且均为**多对多**——一个项目/批次可挂多个协调员，一个协调员可负责多个项目/多个批次。协调员对其关联范围内案件可执行打电话/存证/工单/法务/线下回款标记等。**案件的通话/录音/存证/跟进等数据归属案件、不随协调员人员变动而变化**（BR-M4-10）。
- **BR-M2-14（批次导入与录入）**：批次明细支持 **Excel 导入（主）** 与 **手工逐条新增**；必填字段 = **户号 / 姓名 / 手机号 / 房号 / 应收 / 欠费周期**；导入时**跳过错误行**并报告错误清单（不整批阻断）；**同批次内户号唯一**，重复户号该行报错跳过。**导入/录入权限归物业负责人**（数据源头注入、影响整批，须负责人把关）；**协调员不具备导入权限**（侧重执行：送达 / 工单 / 法务 / 线下回款标记）。导入**按项目逐批次**进行（项目→批次→案件）。**导入字段分两级**：①**必填（影响催收）**=户号/业主姓名/手机号/房号/应收金额/欠费周期，缺失跳过该行；②**选填·诉讼要素（可后补）**=身份证号/建筑面积/通讯地址/其他联系电话/合同编号等，可留空，**生成要素化诉状时再补录**（见 [要素化诉状模板](./要素化诉状模板.md)）。
- **BR-M2-15（要素化起诉对齐）**：项目/批次/案件字段**对齐[要素化诉状模板](./要素化诉状模板.md)**（物业服务合同纠纷）——**项目档案**承载原告（物业公司主体：名称/信用代码/法定代表人/电话/坐落）、合同（类型/附件）、收费标准与周期、违约金等要素；**每户明细**承载被告（业主）/欠费事实要素（周期、金额）。**字段分两级**：①**必填（影响催收）**=户号/姓名/手机号/房号/应收/欠费周期，导入缺失即跳行（BR-M2-14）；②**选填·诉讼要素（可后补）**=身份证号/建筑面积/通讯地址/合同编号等，**可留空，生成诉状时再补录**。目的：为**自动生成要素化起诉状**作数据准备（平台套模板出具 PDF，BR-M4-18）。
- **BR-M2-16（"分期"二义厘清）**：系统中"分期"有两类，**互不等同**：
  - **政策分期**：物业减免规则允许业主分 N 期缴的**正式应收结构**，**改变案件"减免后应收/当期应缴"**，是 **M7 缴费展示与 M8 结清判定**的依据。
  - **承诺分期**：催收员登记的"业主口头承诺分期还款"（M4 BR-M4-13），**仅为履约跟踪、不改变案件应收**；其兑现对照只用于催收策略/信用参考。
  - 即：**结清/应缴只认政策分期；承诺分期不参与应收计算**。
- **BR-M2-18a（减免政策层级 = 阶梯 + 决定权）**：减免政策**项目级为底盘**、**批次级可覆盖**；**案件级不单独设规则**。政策为**阶梯式**，每档含——**折扣比例 / 减免上限（金额）/ 是否含违约金减免 / 决定权**。**决定权两档**：①**催收员自决**——系统直接生效；②**线下内部流程**——系统**仅留痕**、由**物业内部线下处理**（**不在系统内审批**）。
- **BR-M2-18（减免/政策分期的应用与决定权）**：
  - 减免落在**「催收员自决」档**（按减免上限判定）→ **系统直接应用**（自动计算"减免后应收"、生成政策分期计划），**无需审批**。
  - 减免**超出催收员自决档**（落在「线下内部流程」档或超过所有档上限）→ **系统不直接生效**，**仅留痕**并提示**由物业内部线下处理**（不在系统做审批流）。
  - **物业负责人**对任意减免可在**系统内直接核准生效**（减免终点）。
  - 减免政策由**物业负责人**在**创建项目时设置、并在项目档案后期维护**（BR-M2-11）；**批次级可覆盖**。**承诺分期**（M4）不走此流程、不改应收（BR-M2-16）。
- **BR-M2-18b（批次级覆盖与同步 = 作战手册 / 减免政策）**：批次对**作战手册**与**减免政策**各支持两种模式——**继承项目**（默认，项目修改后**自动跟随**）或**自定义覆盖**（独立维护）。**项目级被修改时**：继承中的批次自动跟随；**已覆盖（自定义）的批次保留自定义、不被冲掉**，但系统**标记"项目级已更新·有差异"**，PL 可**查看差异**或**一键同步为项目最新（放弃自定义、重新继承）**。**执行优先级：批次自定义 > 项目级**。
- **BR-M2-17（批次/案件作废）**：误传/录错的**批次或案件可作废**——**仅"待派单"状态**可作废，由**物业负责人**操作（平台超管可代操作）；**物业协调员在其受限的数据范围内（▲）亦可操作**（与权限矩阵 02 §3.2「作废」行 PC=●▲ 对齐），**须填原因、留痕**；**已派单/进行中的不可直接作废**，须走撤案/坏账流程（M8 BR-M8-03）。作废后不参与派单/公海/统计（已计入的聚合相应回退）。

## 4. Epic 与用户故事

### Epic M2-A：项目档案

**US-M2-01 物业负责人创建/编辑项目**
- 作为**物业负责人**，我想创建小区项目、维护各分区属性、并可后续编辑，以便挂批次并给催收提供依据。
- 验收标准：
  - **Given** 我是物业负责人，**When** 我填写基本信息（含**区域**）/合同/缴费标准（描述）/收款信息、**并可指定本项目协调员**后保存，**Then** 项目创建成功并出现在我的项目列表（BR-M2-11）。
  - **Given** 项目已建，**When** 我维护作战手册与减免规则，**Then** 这些内容标记为"催收依据"，将对接单服务商可见（BR-M2-03）。
  - **Given** 项目已建，**When** 我编辑项目档案任意分区，**Then** 修改保存生效；可对项目执行停用（停用后不可再挂新批次，规则 TBD）。
  - **Given** 我是协调员（非负责人），**When** 我尝试编辑项目档案，**Then** 无维护入口/被拒（BR-M2-11）。
  - **Given** 必填分区（基本信息/缴费标准描述）缺失，**When** 我提交，**Then** 系统阻止并提示缺失项。

**US-M2-02 物业负责人挂载协调员（项目级/批次级，多对多）**
- 作为**物业负责人**，我想把协调员关联到项目或批次（可多个），以便其处理对应范围的工单/法务/线下回款/通话/存证。
- 验收标准：
  - **Given** 项目存在且协调员账号已建（M1），**When** 我把**多个**协调员关联到该项目或某批次，**Then** 这些协调员均可在工作台看到对应范围的案件与任务（BR-M2-13）。
  - **Given** 一个协调员，**When** 被关联到多个项目/多个批次，**Then** 其可负责所有关联范围（多对多，BR-M2-13）。
  - **Given** 协调员未关联某项目/批次，**When** 其访问该范围案件，**Then** 无权限。
  - **Given** 某协调员被取消关联或停用，**When** 关联变更，**Then** 案件已产生的通话/录音/存证/跟进数据**仍保留在案件上**，不随之丢失（BR-M2-13、BR-M4-10）。

### Epic M2-B：批次（催收单）

**US-M2-03 物业上传催收批次（Excel 导入 / 手工录入）**
- 作为**物业负责人**，我想用 Excel 导入或手工逐条录入一批欠费案件（催收单），以便交平台撮合催收，并为后期起诉作数据准备。
- 验收标准：
  - **Given** 项目已建，**When** 我用 Excel 导入批次明细（户号/姓名/手机号/房号/应收/欠费周期），**Then** 系统生成批次号并按明细创建案件，状态"待派单"（BR-M2-14）。
  - **Given** 我选择手工录入，**When** 我逐条新增案件明细，**Then** 同样创建案件入该批次（BR-M2-14）。
  - **Given** 明细有缺必填/格式错误行，**When** 导入，**Then** 系统**跳过错误行**并产出错误清单（行号+原因），正确行正常导入（BR-M2-14）。
  - **Given** 同一批次内出现重复户号，**When** 导入，**Then** 重复行报错跳过（BR-M2-14）。
  - **Given** 批次上传成功，**When** 系统记录提交时间，**Then** 以该时间作为 T1 计时起点（BR-M2-04，交 M3）。
  - **Given** 明细字段对齐要素化起诉模板，**When** 案件创建，**Then** 保存起诉所需被告/欠费要素，供后期自动生成起诉状（BR-M2-15）。

**US-M2-07 批次/案件作废（误传纠错）**
- 作为**物业负责人**，我想作废误传的批次或录错的案件，以便纠错。
- 验收标准：
  - **Given** 某批次/案件仍为**"待派单"**，**When** 我作废并填原因，**Then** 其作废、不再参与派单/公海/统计，已计入的聚合相应回退、操作留痕（BR-M2-17）。
  - **Given** 批次/案件**已派单或进行中**，**When** 我尝试作废，**Then** 被拒，提示须走撤案/坏账流程（M8 BR-M8-03）。
  - **Given** 物业无法自行处理，**When** 平台超管代作废，**Then** 生效且留代操作记录（BR-M1-15）。

**US-M2-04 凭批次号直达**
- 作为**任一有权角色**，我想凭批次号直接打开该批次案件列表，以便快速定位。
- 验收标准：
  - **Given** 我有该批次可见权限，**When** 我输入/点击批次号，**Then** 直接进入该批次案件列表，无需逐层下钻（BR-M2-02）。
  - **Given** 我无该批次权限，**When** 我尝试直达，**Then** 无权限且不泄露其存在。

### Epic M2-C：下钻导航与聚合

**US-M2-05 逐层下钻**
- 作为**物业负责人/平台/服务商**，我想从项目→批次→案件逐层下钻并查看聚合，以便掌握进度。
- 验收标准：
  - **Given** 我在项目列表，**When** 我点开某项目，**Then** 看到其批次列表及每批次聚合指标：**案件数 / 应收总额 / 回款额 / 回款率 / 已结清数 / 在催数 / 法务处理数 / 催收进度**（BR-M2-06）。
  - **Given** 我在批次内，**When** 我点开案件，**Then** 看到案件详情（含跟进摘要、池归属、状态）。
  - **Given** 下层案件状态/回款变化，**When** 我查看上层，**Then** 聚合值实时反映（BR-M2-06）。

**US-M2-06 角色化项目视图**
- 作为**服务商**，我想按项目(小区)分组查看接单看板/案件/报表，并查看催收依据，以便高效催收。
- 验收标准：
  - **Given** 服务商接到某项目案件，**When** 我按项目分组浏览，**Then** 案件按项目聚合展示。
  - **Given** 我打开案件，**When** 查看催收依据，**Then** 可见物业↔业主合同/收款信息/物业费标准/作战手册/减免规则（BR-M2-03），不可见物业对账/佣金。
  - **Given** 案件**催收进行中**，**When** 我查看案件，**Then** 催收依据与业主联系方式正常可见（明细可见）。
  - **Given** 案件**已结案**，**When** 我（服务商/催收员）查看，**Then** **只见统计、不见案件明细**（业主信息/联系方式/跟进/录音不可见）（BR-M1-06、M8 BR-M8-09）。
