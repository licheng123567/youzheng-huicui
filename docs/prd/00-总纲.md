## 1. 产品定位

**一句话**：有证慧催是一个 **SaaS 物业费催收的居间撮合平台**。

物业把欠费案件交给平台 → 平台撮合给服务商催收（由服务商旗下催收员执行）→ 催收员通过电话催收，系统自动录音 → 平台解析录音转文本，并以 AI 能力对录音/通话提供话术辅助 → 全过程留痕并支持区块链存证。

- **催收场景**：物业费催收（小区物业费欠缴）。
- **平台不直接催收**，只做撮合、能力供给（录音转写 / AI 话术 / 存证 / 法律服务）、规则与结算。
- **护城河（三层）**：
  1. **AI 话术能力**：通过 ASR 把通话转文本，分析"哪些话术更有效"，并在通话前依据**用户画像**给出话术建议与催收策略（沉淀为质检 + 培训闭环）。
  2. **存证能力**：提供全过程存证，**为后期诉讼提供法律证据**。
  3. **平台法律服务**：平台提供法律服务能力，如**仲裁、律师函**等。

## 2. 商业模式与资金流向

**佣金（平台 ← 物业）**
- 平台与**物业公司**签约，向物业公司**收取佣金**。
- 佣金 = **回款 × 佣金比例**；**比例由平台与物业公司协商确定，按批次（催收单）管理**（不同批次可不同比例）。
- 平台向**服务商**支付佣金。
- **服务商与物业之间不产生任何经济往来**（强约束，贯穿结算与权限设计）。

**能力供给与计费**
- 平台向物业 + 服务商提供：录音转文字（STT）、AI 话术辅助。
- **录音转文字按分钟计费**（计量单位：**通话解析分钟数**，无"分值"中间概念）：**哪个组织使用，就由哪个组织付费**；通过在平台**预充通话解析分钟数**、按使用扣减。**计费的是"录音上传平台做语音解析(STT+策略)"环节，打电话+录音本身不受余额限制**；余额不足只暂停自动解析、可充值后手动补解析。
- **存证 与 法律服务（仲裁 / 律师函）只向物业公司收费**（不向服务商收）。
  - **法律服务由平台出具**（如仲裁、律师函），**物业协调员负责线下送达、签收拍照**回传。
  - **存证三场景**（M6）：①法务文书送达存证、②通话录音存证（**一般录音不存证，仅协调员发起法务后由协调员决定**）、③证据材料打包存证（法务处理中协调员打包催收+沟通记录时决定）；**线下回款凭证不存证**。

```
                       佣金(收)             佣金(付)
        物业公司  ───────────────►  平台  ───────────────►  服务商
          ▲                          │                        │
          │ 物业费回款                │ 能力供给                │ 催收执行
          │ (业主→物业)              ▼ (转写/AI/存证)          ▼
        业主  ◄──────── 催收触达 ──────────────────────── 催收员
   ※ 物业 ↔ 服务商：无任何资金往来
```

> 具体佣金口径、STT 单价等数字见 §8 可配置参数表，本轮留 TBD。

## 4. 角色总览

平台为典型多租户 SaaS，三方组织，每方为「单一负责人 + N 个员工」结构（平台侧为"平台超管 + 平台员工"），按功能授权。

| 方 | 角色 | 职责摘要 |
|---|---|---|
| **平台侧** | 平台管理员（超管） | 全局配置、撮合派单、公海处置、参数配置、对账、报表、代物业结案操作 |
| | 平台员工 | 由超管按用户体系授权相关功能（子集） |
| **物业侧** | 物业负责人 | 建项目、加催收批次、减免政策、与平台对账、查看催收进度、加协调员、决定撤案/坏账结案；**充值通话解析分钟数**；可用 App 给案件联系人打电话（自动录音转写、**关联案件跟进记录**，但**不驱动公海/派单**） |
| | 物业协调员 | 标记线下回款、处理工单回执、**独立主导法务（催收函/律师函/起诉状）**、缴费链接；**可直接撤案/坏账结案（须选原因）**；**用 App 打电话（自动录音转写、关联案件跟进记录，不驱动公海/派单）**；**送达签收拍照上传**；**存证管理（①送达存证 ②录音存证-发起法务后协调员决定 ③证据材料打包存证）** |
| **服务商侧** | 服务商负责人 | 案件分配、与平台对账、**充值通话解析分钟数**、加催收员 |
| | 催收员 | 通话催收、转工单要求、发缴费链接、标注跟进 |

> 角色与权限的完整模型见 **M1 组织与权限**。

## 5. 核心对象与导航主轴

**项目(小区) → 批次(催收单) → 案件** 是贯穿全平台的**导航、筛选、聚合主轴**。

- 任意层级可逐层下钻；也可凭**批次号**独立直达某批次的案件列表。
- 报表（收入+支出+利润）按项目聚合并可下钻到批次，或独立按单批次出。

| 对象 | 定义 | 上级 |
|---|---|---|
| **项目（小区）** | 一个物业小区的档案，含合同/缴费标准/收款信息/协调员/作战手册/减免规则 | 物业公司 |
| **批次（催收单）** | 物业一次性提交的一组欠费案件（按项目挂载） | 项目 |
| **案件** | 单个欠费业主/户的催收对象（含应收、状态、归属、跟进记录） | 批次 |

> **案件唯一性规则**：同一业主在不同批次/缴费周期重复欠费，**视为新案件，仅批次不同**——案件归属于唯一批次，**不跨批次累加/合并**。同一业主因此可在系统中对应多个案件（分属不同批次），各自独立流转、催收、结案。

> 主数据字段与下钻交互见 **M2 项目-批次-案件主数据**。

## 6. 案件全生命周期（状态总览）

下图为**案件状态总览**，各状态的进入/退出条件、时效、护栏细节分别在 M3（流转/公海）与 M8（结案/生命周期）展开。

```
[待派单] ──撮合派服务商──► [服务商公海] ──抢单/负责人分配──► [私海·进行中]
   │T1超时                      │T2滞留超时                    │ T_collector 无跟进自动释放
   ▼                            ▼                             ▼ (放回对应公海, 仍"进行中")
[平台公海] ◄────────────────────┘                    [私海·进行中] 持续催收
   │ 平台二选一: (A)再派其他服务商 / (B)开放抢单
   ▼
 ... 回到对应公海/私海

进行中案件的终态:
  • 应收全部到账 ─► [已结清]   (系统自动结案)
  • 协调员/负责人决定 ─► [已撤案/坏账]  (须选原因, 平台超管可代操作)
结案后 ─► 服务商/催收员只见统计、不见明细；案件明细仅平台+物业可见
```

**关键不变量（贯穿全局）**：
- **释放回公海 ≠ 结案**：案件仍为"进行中"。
- **催收员/服务商不能结案**：只能释放回公海、标注跟进、打"建议法务"标注（法务由协调员独立主导，与催收并行；不做法务硬交接）。
- **结清**由系统自动判定；**撤案/坏账**由物业协调员或物业负责人直接操作（须选原因，平台超管可代操作）。
- **通话记录全局关联案件**：无论哪个角色（催收员 / 物业负责人 / 协调员）通过 App 给案件联系人打电话，均**自动录音 + 转写**，并作为该案件的**跟进记录**关联；案件**关联人（有权者）均可回放录音、查看文本、获取转写后的下一步策略**。消耗使用方组织的通话解析分钟数。
- **物业侧通话不驱动池流转**：物业负责人/协调员打电话**不走公海、无派单流程、不改变案件池归属**；案件的公海/派单/抢单/释放仅由**服务商侧催收**驱动。
  - **已定**：T_collector **仅由持有催收员本人的跟进重置**；物业负责人/协调员的通话进案件时间线、可见，但**不重置** T_collector（M3 BR-M3-08）。

## 7. 全局术语表

| 术语 | 定义 |
|---|---|
| 平台公海 | 兜底/再分配池：派单时限 T1 内未派单的案件 + 服务商公海滞留 T2 超时退回的案件 |
| 服务商公海 | 已派给某服务商、供其旗下催收员抢单的案件池 |
| 私海 | 已被某催收员持有的案件，**不属于公海** |
| 开放抢单 | 平台对平台公海案件的处置之一：标记后全平台催收员均可抢，抢中后归入其所属服务商结算（不设在岗/离岗概念） |
| 跟进记录 | 通话 / 标注 / 小结 / 承诺 等任一催收动作 |
| 派单时限 T1 | 物业上传后，平台须在此时限内撮合派单，超时入平台公海 |
| 服务商处置时限 T2 | 案件进服务商公海后须在此时限内被分配/抢单，超时退回平台公海 |
| 催收员无跟进释放 T_collector | 催收员持有案件后连续此天数无新增跟进记录，自动释放回公海 |
| 最长催收周期 | 案件入池后允许的最长催收时长，超时提醒物业负责人处置（不自动结案） |
| 结案可见性收敛 | 案件结案后，**服务商与催收员只看统计、不看明细**；案件明细仅平台+物业可见（无过渡期，M8 BR-M8-09） |
| 再派护栏 | 平台再派"其他"服务商时必须满足的 2 项实判：①≠原退回服务商 ②目标服务商启用中（详见 M3 §3.6） |
| STT | 录音转文字（Speech-to-Text），按分钟计费 |
| 作战手册 | 项目级催收依据/话术指引 |
| 通话解析分钟数 | STT 计费的计量单位；任一组织用 App 通话录音转写均消耗，预充值、按使用扣减（**无"分值"中间概念**） |
| 跟进记录（通话类） | 任一角色对案件联系人的 App 通话均自动录音+转写并关联为该案件跟进记录，案件关联人可回放/看文本/取策略 |
| 存证 | 关键证据上链留痕、不可篡改，为后期诉讼提供法律证据（M6）；**只向物业收费** |
| 送达存证 | 存证场景①：法务文书（催收单/仲裁书/律师函）线下送达的签收拍照等留痕上链 |
| 通话录音存证 | 存证场景②：**一般录音不存证**；仅**协调员发起法务后由协调员决定**是否对关键录音存证（M6 BR-M6-02） |
| 证据材料打包存证 | 存证场景③：法务处理中协调员打包下载催收+沟通记录为证据材料时，决定是否上链存证（M6） |
| 法律服务 | 平台出具的法律文书/动作（仲裁、律师函等），协调员负责线下送达签收拍照；只向物业收费 |

## 9. 全局非功能性约束（轻量登记，本轮不展开实现）

|---|---|
| 多租户隔离 | 物业只见自己项目；服务商只见派给自己的案件；催收员只见本服务商池+自己私海（除开放抢单） |

| 全程留痕 | 通话录音、跟进记录、状态变更、派单/抢单/释放均留操作日志 |
