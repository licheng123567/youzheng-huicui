# 有证慧催 PRD · M3 案件流转与公海

> 依赖：[00-总纲](./00-总纲.md) §6 生命周期、§7 术语、§8 CFG 参数。
> 目标：定义案件在 **平台公海 / 服务商公海 / 私海** 间的完整流转：派单 T1、抢单、分配、释放 T_collector、退回 T2、平台二选一处置（再派/开放抢单）、再派护栏、并发安全、实时竞争（公海无可配动态规则）。
> 本模块是产品骨架，规则最密集。
> **契约对齐**：本模块与 API 契约 v1.0.0 对齐。

---

## 1. 三类池定义

| 池 | 定义 | 谁可见/可操作 |
|---|---|---|
| **平台公海** | 兜底/再分配池：T1 内未派单的案件 + 服务商公海滞留 T2 超时退回的案件 | 平台运营撮合/处置；标记"开放抢单"后对全平台催收员可见 |
| **服务商公海** | 已派给某服务商、供其旗下催收员抢单的案件池 | 该服务商负责人（可分配）、该服务商催收员（可抢） |
| **私海** | 已被某催收员持有的案件，**不属于公海** | 持有催收员（催收作业）、其服务商负责人（可见） |

## 3. 业务规则

### 3.1 派单（T1）
- **BR-M3-01**：批次提交后进入"待派单"，由平台运营在 **T1（CFG-T1，TBD）** 内撮合派给某服务商。**派单粒度可整批派、也可按单案件派**（同一批次的案件可拆派给不同服务商）。
- **BR-M3-02**：T1 内未派单 → 案件自动进入**平台公海**（兜底）。
- **BR-M3-03**：成功派单后案件进入该服务商的**「待接单」**，经服务商承接后入**服务商公海**（BR-M3-03a）。
- **BR-M3-03a（服务商承接/接单）**：平台派单（整批/拆单）到某服务商后，案件先进入该服务商**「待接单」**，由**服务商负责人**处理：**接单（承接）→ 进服务商公海待分配给催收员**；**拒接 → 批次退回平台公海由平台重派**（拒接须填原因、留痕，口径同主动退案 BR-M3-28，计入客观指标 BR-M3-24）。承接后即占该服务商结算归属与持有口径。

### 3.2 抢单 / 分配
- **BR-M3-04（抢单范围）**：催收员**默认只抢本服务商公海**；仅当平台把案件标记**「开放抢单」**后，才可跨服务商抢平台公海开放池案件；**抢中后归入其所属服务商结算**。
- **BR-M3-05**：服务商负责人可把本服务商公海案件**直接分配**给本服务商催收员（无需抢）；**分配同样受持有上限约束**（目标催收员达上限则不可分配，见 BR-M3-06）。
- **BR-M3-06（持有上限）**：单催收员私海持有上限 **CFG-HOLDCAP（平台可配，TBD）**；达上限不可再抢/被分配（对抢单与负责人分配均适用）。
- **BR-M3-07（并发安全）**：抢单须防并发——同一案件不可被两人同时抢中（行级锁/原子占用，对服务商公海与开放抢单均适用）。

### 3.3 释放（T_collector）
- **BR-M3-08**：催收员持有后**连续 T_collector（CFG-TC，TBD）该持有催收员本人无新增跟进记录** → 系统自动释放（定时任务）。跟进记录 = 该催收员的通话/标注/小结/承诺等任一动作。
  - **T_collector 仅度量持有催收员本人的跟进**：其他角色（物业负责人/协调员）对该案件的通话等动作会记入案件时间线、案件关联人可见，但**不重置 T_collector**（结 OQ-6）。
  - T_collector 计时**按自然日**（含周末/节假日连续计；不做工作日模式，OQ-M3-3 已定）。
  - **存在进行中有效分期承诺时暂缓 T_collector 自动释放**（由承诺到期预警驱动跟进；某期违约则恢复计时）——见 M8 BR-M8-11。
- **BR-M3-09（释放去向）**：服务商公海来源的案件 → 回**服务商公海**；开放抢单来源的案件 → 回**平台公海开放池**。
- **BR-M3-10**：**释放即单纯放回公海，无佣金奖惩**；**释放回公海 ≠ 结案**，案件仍"进行中"。
- **BR-M3-11**：催收员可**主动释放**私海案件回来源公海（同样无奖惩、不结案）。
- **BR-M3-12**：账号停用（M1 BR-M1-07）触发其私海案件释放回来源公海。

### 3.4 服务商公海退回（T2）
- **BR-M3-13**：案件在服务商公海滞留 **T2（CFG-T2，TBD）** 未被分配/未被抢 → 自动退回**平台公海**。退回后该服务商即成为"原退回服务商"，平台再派受护栏①约束（BR-M3-16）。
- **BR-M3-13a（T2 预警）**：案件**临近 T2**（预警提前量可配，TBD）时，系统向**服务商负责人**预警"某案即将退回平台公海"，以便其及时分配/促抢。预警不阻止到期退回。

### 3.5 平台公海二选一处置
平台对平台公海案件二选一：
- **BR-M3-14（A 再派其他服务商）**：可再派给**其他**服务商，须通过**再派护栏**（§3.6）；通过后进入新服务商公海。
- **BR-M3-15（B 开放抢单）**：标记「开放抢单」→ 进入平台公海开放池，**全平台催收员均可抢、面向所有服务商全面开放（不设白名单/服务商范围限定）**（**不设在岗/离岗概念，能抢即视为在岗**）；抢中归入其所属服务商结算，并占用其持有上限。**前置：标记开放抢单前平台须先为该批次设"开放抢单付佣比例"（M9 BR-M9-18），未设不可开放**。

### 3.6 再派护栏（再派"其他"服务商时必须满足，2 条实判）
- **BR-M3-16（护栏①≠原退回方）**：再派的服务商 ≠ 该案件曾被退回的来源服务商（不回踢给原服务商）。**若案件因 T1 超时进平台公海（从未派过任何服务商），则无"原服务商"，本护栏不适用，可派任意启用中服务商**。
- **BR-M3-17（护栏②服务商启用中）**：再派目标服务商须为**"启用"状态**（未被平台停用/拉黑），即具备接单资格。
- **BR-M3-18（隔离恒成立，无需单列护栏）**：派单只改案件归属、不产生物业↔服务商资金往来，**隔离天然成立**，故不单列资金/隔离护栏。实质护栏即上述 2 条。

### 3.6′ 时效参数
- **BR-M3-19**：T1、T_collector、T2、最长催收周期均由**平台超管可配**（§8 CFG）。**参数变更只对变更后新进入计时的案件生效；存量在途案件沿用其开始计时时的旧值**（不重算）。

### 3.7 实时竞争 + 内海池（公海无可配动态规则）
- **BR-M3-20（公海不设可配动态规则）**：本产品公海**不设可配动态规则**——**无抢单窗口、无佣金加权激励、无新人优先级**（保持最简）。公海仅提供：实时竞争态展示、入池来源徽标、实时事件日志。
- **BR-M3-21（实时竞争）**：案件展示实时抢单竞争态——待抢 / 正在查看 N 人 / 已抢；并带**入池来源徽标**（T1超时入池 / T2退回 / 开放抢单等）。
- **BR-M3-21a（未持有前联系方式脱敏）**：公海池（服务商公海 / 平台公海开放池）中**尚未被持有/抢中的案件**，其**联系电话脱敏展示**（如 138\*\*\*\*0001），仅显业主姓名 / 房号 / 应收 / 状态 / 入池来源等用于判断的字段；**抢中（进入私海）后**，持有催收员方可见完整号码并外呼（呼应 M4 BR-M4-01b）。结案后仍按 M8 BR-M8-09 收敛。
- **BR-M3-22（公海面板）**：公海页含**实时事件日志**面板（**无规则配置入口、不含佣金加权排行榜**；如需排行仅为简单抢单计数统计，可选）。
- **BR-M3-23（服务商内海池）**：服务商侧显示距自动转平台倒计时（T2）、各催收员持仓余量条、按余量智能推荐分配。
- **BR-M3-24（派单决策辅助）**：平台撮合派单时可查看各服务商的**客观经营指标**（在催案件数、催收员数、人均持仓、近期回款表现等）以辅助决策。**不做"评价/权重算法"**（不引入物业评价、不把风险率纳入派单；风险率仅用于 M5 质检/风控）。
- **BR-M3-25（分配粒度）**：**一个案件只能分配给一个催收员**（私海单一持有）；服务商负责人**可批量把多件案件分配给同一个催收员**（受持有上限约束，超限部分拒绝）；**"把 N 件平均分给一组催收员"为可选增强**（实现简单则纳入，否则后置）。
- **BR-M3-26（释放预警）**：自动释放前，**临近 T_collector**（预警提前量阈值可配，TBD）系统向持有催收员预警"某案即将释放"；催收员**新增任一跟进即可续持**（重置 T_collector）。预警不阻止到期释放。
- **BR-M3-27（主动释放须填原因 + 可见）**：催收员主动释放**须选择原因**（平台默认枚举：**无法联系 / 非本人 / 其他**；后期可配；+ 可选备注）；**所有释放行为（主动/自动）均留记录**，**服务商负责人可见本服务商催收员的释放历史与频次**，用于管理。释放记录不改变佣金奖惩（BR-M3-10）。
- **BR-M3-28（服务商主动退案）**：服务商负责人可对本服务商公海/私海中**不愿承接的案件主动退回平台公海**（可整批或按单案件），**须填原因、留痕**；退回后该服务商即成为该案件的"原退回服务商"，平台再派受护栏①约束（BR-M3-16）。**平台可见各服务商主动退案频次/原因**作为派单参考（计入 BR-M3-24 客观指标）。主动退案不改变佣金（无奖惩）。

## 4. 案件池状态机

```
待派单 ──T1内派单──► 服务商公海 ──抢单/分配──► 私海(进行中)
   │T1超时              │T2滞留超时              │ 主动释放 / T_collector自动释放
   ▼                    ▼                        ▼
平台公海 ◄──────────────┘              回来源公海(服务商公海 或 平台公海开放池)
   │
   ├─(A)再派其他服务商(过再派护栏)─► 新服务商公海
   └─(B)开放抢单─► 平台公海开放池 ──任一催收员抢──► 私海(归其服务商结算)
```

## 5. Epic 与用户故事

### Epic M3-A：派单与平台公海兜底

**US-M3-01 平台运营撮合派单**
- 作为**平台运营/超管**，我想把待派单批次/案件按项目分组撮合派给服务商，以便启动催收。
- 验收标准：
  - **Given** 批次待派单且在 T1 内，**When** 我选定服务商整批派单，**Then** 案件进入该服务商公海，状态"已派单"，并记录派单方/时间（BR-M3-03）。
  - **Given** 一个批次内多个案件，**When** 我勾选部分案件派给服务商 A、其余派给服务商 B，**Then** 各部分分别进入对应服务商公海，同批次案件可分属不同服务商（BR-M3-01）。
  - **Given** 我在派单，**When** 我查看候选服务商，**Then** 可见各服务商当前负荷/在催量等指标辅助决策（BR-M3-24）。
  - **Given** 案件提交已超 T1 仍未派单，**When** T1 到期，**Then** 案件自动进入平台公海并打"T1超时"入池徽标（BR-M3-02、BR-M3-21）。
  - **Given** 我在撮合页，**When** 我按项目分组浏览待撮合批次，**Then** 同项目批次聚合展示。

**US-M3-02 平台公海二选一处置**
- 作为**平台运营/超管**，我想对平台公海案件选择"再派其他服务商"或"开放抢单"，以便兜底再分配。
- 验收标准：
  - **Given** 平台公海有一被服务商 X 退回的案件，**When** 我选"再派"并选择服务商 Y，**Then** 系统校验再派护栏（Y≠X 等），通过则进 Y 的服务商公海（BR-M3-14、BR-M3-16）。
  - **Given** 案件因 **T1 超时**进平台公海（从未派过任何服务商，无原服务商），**When** 我再派任意有效服务商，**Then** 护栏①不适用，派单成功（BR-M3-16）。
  - **Given** 我选择的目标服务商不满足护栏（=原退回服务商 / 已停用），**When** 我提交，**Then** 系统拒绝并提示具体护栏原因。
  - **Given** 平台公海案件，**When** 我标记"开放抢单"，**Then** 案件进入开放池，全平台催收员可见可抢（BR-M3-15）。

### Epic M3-B：抢单与分配

**US-M3-03 催收员抢本服务商公海案件**
- 作为**催收员**，我想从本服务商公海抢单到私海，以便开始催收。
- 验收标准：
  - **Given** 本服务商公海有可抢案件且我持有数 < 上限，**When** 我点击抢单，**Then** 案件进入我的私海，状态"进行中"，归属本服务商（BR-M3-04）。
  - **Given** 我持有数已达上限（CFG-HOLDCAP，平台可配），**When** 我抢单，**Then** 被拒并提示已达持有上限（BR-M3-06）。
  - **Given** 同一案件被两名催收员几乎同时点抢，**When** 并发到达，**Then** 仅一人成功，另一人收到"已被抢"提示（BR-M3-07）。
  - **Given** 案件未被标记开放抢单，**When** 其他服务商催收员尝试抢，**Then** 不可见/不可抢（BR-M3-04）。

**US-M3-04 跨服务商抢"开放抢单"案件**
- 作为**任一催收员**，我想抢平台公海开放池案件，以便扩大可催收量。
- 验收标准：
  - **Given** 案件已被平台标记"开放抢单"，**When** 我抢单，**Then** 案件进我的私海，并标记"归入我所属服务商结算"（BR-M3-15）。
  - **Given** 抢中开放案件，**When** 进我私海，**Then** **占用我的持有上限**（CFG-HOLDCAP）；若已达上限则不可抢（BR-M3-06）。
  - **Given** 并发抢同一开放案件，**When** 多人同时，**Then** 仅一人成功（BR-M3-07）。

**US-M3-05 服务商负责人分配案件**
- 作为**服务商负责人**，我想把本服务商公海案件分配给指定催收员，并按余量智能推荐，以便均衡产能。
- 验收标准：
  - **Given** 服务商公海有案件且目标催收员未达上限，**When** 我分配，**Then** 案件进该催收员私海（一案一人，BR-M3-05/25）。
  - **Given** 我勾选多件案件，**When** 批量分配给同一催收员，**Then** 在其剩余额度内全部进入其私海；超出额度的部分被拒并提示（BR-M3-25/06）。
  - **Given** 我打开分配面板，**When** 查看催收员持仓余量条，**Then** 系统按余量给出智能推荐分配（BR-M3-23）。
  - **Given**（可选增强）我选 N 件并选定一组催收员，**When** 选择"平均分配"，**Then** 系统在各人额度内尽量平均分派（BR-M3-25，可选）。
  - **Given** 目标催收员已达上限，**When** 我分配，**Then** 被拒并提示。

### Epic M3-C：释放与退回（时效驱动）

**US-M3-06 催收员无跟进自动释放**
- 作为**平台（系统）**，我想在催收员连续 T_collector 天无新增跟进时自动释放案件，以便案件不被沉睡占用。
- 验收标准：
  - **Given** 催收员持有案件且连续 T_collector 天无任何跟进记录，**When** 释放任务运行，**Then** 案件按来源释放回公海（BR-M3-08、BR-M3-09），状态仍"进行中"（BR-M3-10）。
  - **Given** 案件来源为服务商公海，**When** 自动释放，**Then** 回该服务商公海；**Given** 来源为开放抢单，**When** 自动释放，**Then** 回平台公海开放池（BR-M3-09）。
  - **Given** 释放发生，**When** 我查看佣金/绩效，**Then** 无任何奖惩变化（BR-M3-10）。
  - **Given** 催收员在第 T_collector 天前新增了一条跟进（通话/标注/小结/承诺任一），**When** 任务运行，**Then** 不释放，计时重置。
