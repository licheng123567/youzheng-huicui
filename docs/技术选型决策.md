## TD-1　通话录音管线（一期）

**决策**：App 仅作催收员工作列表，核心三件事——①选案件 → 自动调起**原生报号器拨号**（号码取自案件主数据，不手输）；②系统录音 → 录音文件**事后异步上传云端解析**（非实时）；③角色相关移动功能（如协调员送达拍照存证）。录音机制采用 **(A) 读系统通话录音文件夹**（依赖国内 OEM ROM 自带"自动通话录音"，可录到双方声音，已实验验证）。

**理由 / 取舍**
- **用真实手机号打电话 = 解决业主信任**：呼叫中心号码易被标记骚扰、接通率低。一期不上呼叫中心。
- AOSP 原生第三方应用录不到对方声音（Android 9/10+ 限制），但**国内 OEM ROM 的系统通话录音在系统层可录双方** → 走读系统录音目录(A)，**不用** App 自录(MediaRecorder，碎片化严重)。
- 事后异步解析匹配 BR-M5-02（通话后解析、非实时）与"App 离线暂存+恢复同步"。

**落地要点**
- `FileObserver` 监听录音目录 `CLOSE_WRITE`（文件写完才上传）；录音目录因 ROM 而异，按 ROM 登记。
- 按"号码 + 通话起止时间"把录音文件回填到对应案件，关联为跟进记录。
- Android 11+ 分区存储下读系统录音目录需引导授权（`MANAGE_EXTERNAL_STORAGE` 或目录授权）。
- **机型白名单 + App 启动自检**：检测 ROM 是否开启自动通话录音 / 目录是否可读 / 能否拿到双向音；不通过则引导设置或提示换机。后台维护「机型 + ROM 版本 + 录音目录 + 是否验证」矩阵。**固有边界**：无此功能的 ROM 落白名单外。
- **已接通 / 通话时长靠录音反推**（原生通话对第三方应用是黑盒）：时长 ≈ 录音文件时长；接通 = 有无录音文件 + VAD 是否检出双方有效语音 → 喂 M4 `CFG-MARK-CODES`（是否接通/有效跟进）。
- **计费口径不变**：打电话+录音不耗分钟，**上传解析才按分钟扣**（`CFG-STT-PRICE`）；余额不足暂停自动解析、可手动补解析（BR-M5-01/02）。

**二期演进**：呼叫中心（FreeSWITCH 自建 / 云通信）实现实时采集 + 实时分析。二期天然**双声道**，说话人分离从"算法猜"变"分声道"自动变准。（OQ-3 据此收敛为"软电话/系统录音"路线。）

## TD-2　ASR 选型

**决策**：阿里百炼（Model Studio / DashScope）**录音文件识别**，模型 **`paraformer-8k-v2`**。

**理由**
- 专为 **8k 电话采样率**优化，匹配手机原生通话录音音质。
- 同时支持四项刚需：**热词**（治错字）、**说话人分离**、**句级时间戳**（支撑 BR-M5-11 气泡点击定位）、敏感词过滤（辅助 M5 风控）。
- Qwen-ASR 因**不支持热词/说话人分离**被排除；Fun-ASR 作备选/对比（时间戳固定开启）。
- **关键利好**：阿里"说话人分离仅支持**单声道**音频" —— 正合原生混音单轨录音；调用传 `diarizationEnabled:true` + `speakerCount:2`，按句返回 `speaker_id`。说话人分离是**调用传参**非控制台开关；热词先建表拿 `vocabularyId` 再传入。
- **迁移红利**：百炼 Fun-ASR/Paraformer = 达摩院开源 **FunASR** 云托管版，二期自建降本时迁开源 FunASR，模型同源、行为一致。

**热词自动生成**（项目级，拨号附带）
| 来源 | 内容 | 取自 |
|---|---|---|
| 业主姓名 | 项目在催业主名（最易识错） | M2 案件 |
| 小区/楼栋/项目名 | 小区名、楼栋号、门牌格式 | M2 项目/批次 |
| 领域术语（静态词典） | 物业费/滞纳金/违约金/公摊/电梯费/停车费/欠费周期/减免/分期/催缴/起诉/仲裁/律师函… | 平台维护 |
- 条数超百炼上限时降批次级 / call 级（只注本案业主名 + 号码标签家属名）。零 LLM 成本，**优先于 LLM 纠错**。

**纠错四层（性价比序）**：①热词（解决多数错字）→ ②8k 大模型档位 → ③LLM 同音字纠错后处理（Flash）→ ④音频预处理（降噪/增益）。
**红线**：金额/日期/分期期数 ASR 错危害最大（喂承诺跟踪 / 回款标签 / 风控）→ **置信度标记 + LLM 重点校验**，必要时催收员二次确认。

**计费**：单价查百炼定价页，登记 `CFG-STT-PRICE`，按分钟扣。

## TD-3　LLM 选型与模型分档

**决策**：**DeepSeek 官方直连**，按管线分档（不一刀切）。

| 环节 | 档位 | DeepSeek 型号 |
|---|---|---|
| ① 通话结构化抽取（出 JSON） | Flash（便宜走量） | `deepseek-chat`（V3） |
| ③ ASR 纠错 / ⑥ 风险分类 | Flash | `deepseek-chat` |
| ④ 逐通话策略建议 | Pro / 推理版（要质量） | `deepseek-reasoner`（R1） |

- **合规硬约束（BR-M5-09）**：业主敏感信息发往 DeepSeek 前**脱敏占位**（如 `业主A`/`138****`），确认其数据使用政策；飞轮语料留存在脱敏后。

## TD-4　护城河与飞轮设计

**护城河 ≠ LLM**（模型大家都能买）。护城河 = ①物业催收专有「转写+结构化结果+回款结果」标注语料；②飞轮数据网络效应（BR-M5-12）；③话术库结构性锁定（只输出建议、不吐库本体 BR-M5-06）；④风控/质检合规层粘性。→ 推论：**录音/ASR/语料必须归平台所有，不长期外租**。

**六段管线**
```
ASR转写(speaker_id+时间戳)
  ① 每通结构化抽取(Flash) → call_analysis 原子记录
  ② 话术片段切片(按场景) → utterance_snippet + 结果标签
  ③ 有效性归因(数据+统计,非纯LLM) → 平台话术库 playbook_snippet
  ④ 策略建议 = RAG(话术库)+画像+历史 → R1 改写成"建议"(不吐库本体)
  ⑤ 作战手册初稿 = 聚合+物业录入 → R1 → 物业负责人审核采纳 → 版本化/增量重生成
  ⑥ 风险检测(全量) = 脏话词表规则高召回前置 → 命中再上 Flash 判语义
```

**飞轮核心：承诺桥归因**（因果链 `话术 → 承诺 → 兑现(回款)`）
- **短期信号**（通话结束即有，来自本通话 call_analysis）：接通 / 拿到承诺 / 情绪好转（开头 vs 结尾）。
- **中期信号**（数天后到达，来自 M4 回款登记 / M9 回款时点）：承诺兑现 / 是否回款 / 回款时长。
- **join 算法**：对有承诺的通话，在同案件回款记录里按 `|回款时间 - 承诺到期| ≤ 窗口` 且 `金额≈承诺额` 匹配 → 标"承诺兑现"、回填片段标签；无承诺直接回款用"时间窗最后一通"低权兜底。
- 比时间衰减归因干净 —— **承诺是显式、被系统登记的中间结果**（M4 已有承诺登记）。

**画像 cohort 分层**（否则不可比）
- 统计维度 = `场景(scene) × 话术意图(intent_tag) × 画像cohort`。
- cohort 分桶 = 欠费金额档 × 账龄档 × 历史态度 × 是否争议。
- 支撑 BR-M5-12"相似画像下话术 A/B 对比"。画像字段源 = M2 案件 + 历史 call_analysis 聚合（回应 OQ-M5-5）。

**入库门槛（防小样本误判）**：`N ≥ 阈值` 且相对 cohort 基线显著提升才从候选池沉淀进话术库；小 N 用 Wilson 下界/贝叶斯平滑；低效标记淘汰。

**原子数据单元**
- `call_analysis`（每通一条 JSON）：场景/业主态度/争议点/承诺{金额,到期,分期}/异议/情绪轨迹/风险命中/下一步种子。
- `utterance_snippet`（话术片段）：场景/意图标签/脱敏文本/时间区间/催收员 + join 回填结果标签。
- `playbook_snippet`（平台话术库，平台级跨项目）：**存泛化模板**（剥离 PII/项目特征，跨项目可复用且不泄库本体）；④ 靠 RAG 检索模板 + R1 按当前案件重新实例化，对外只输出"建议"。

**更新节奏（BR-M5-12）**：①抽取/⑥风控=每通实时（Flash）；②切片/③归因/话术库=**批量增量**（中期标签晚到 + 统计累积，夜跑/阈值触发）；④策略=每通实时（用库快照）；⑤手册=物业触发增量重生成。

**冷启动**：种子话术库（物业录入情况 + 人工标注数十通优质通话 + 通用催收话术）→ 先规则/人工 → N 够再切统计。作战手册初稿需最小通话量（OQ-M5-4）。

## TD-5　技术底座：统一 Java 单栈

**决策**：**纯 Java 单栈**（Spring Boot + 若依 RuoYi），**不弄两套**；核心域（撮合/公海/案件状态机/计费/隔离）自建，**不拿通用 CRM 当底座**。

**理由（Java vs Python 二选一）**
1. 系统 90% 是**事务型 SaaS**（多租户隔离/RBAC双维度授权/案件状态机/并发抢单防重/分钟·条数·次预付费扣减/对账/存证留痕）——Java/Spring 主场；M5 AI 只是其中一个模块。
2. AI 部分本质是**调云 API**（DeepSeek=OpenAI兼容、百炼=官方 Java SDK），Java 直调无障碍，**不为调 AI 引入 Python**。
3. 飞轮统计主要是 **SQL 聚合 + 业务逻辑**（承诺桥/cohort），RAG=pgvector Java 客户端，未到需 pandas/sklearn。
4. **若依直接落地 M1**：RBAC/多租户/菜单权限/操作审计/代码生成开箱即用。国内 Java SaaS 后台人才池大。

**不拿 CRM 当底座**：通用 CRM（SuiteCRM/Vtiger/Odoo/EspoCRM）是销售管道模型，不匹配「项目→批次→案件 + 三方撮合 + 三类租户隔离」。

**Python 的唯一去处**：**二期自建 ASR/LLM（FunASR/Qwen 上 GPU）时的模型推理薄微服务**，藏接口后被 Java 调 —— 业务系统仍 100% Java，**不算两套底座**，只是换供应商实现。飞轮若上重型 ML 同理加 Python 批处理 worker（附加件，非并行系统）。
**翻盘条件**：若团队实为 Python 为主、Java 很弱，则反过来纯 Python（团队熟练度 > 契合度边际差）。当前团队两者皆可 → 按系统性质选 Java。

## TD-6　周边组件 + 总体架构

| 层 | 选型 | 备注 |
|---|---|---|
| 语言/框架 | Java + Spring Boot | |

| Python（仅二期触发） | 自建模型推理薄微服务 | 被 Java 调 |

**总体架构（一期）**
```
App(原生拨号+读系统录音目录) ──上传──► OSS(录音)
                                          │
            ┌─────────────────────────────┘
            ▼  (异步, 消息队列触发)
[ASR: 百炼 paraformer-8k-v2 + 热词(取自案件)] ──► 转写(speaker_id+时间戳)
            │
[① 抽取 / ② 切片 / ⑥ 风控  (deepseek-chat)] ──► call_analysis / snippet 落 PostgreSQL
            │
            ├──事件{承诺,风险}──► [Java 业务: 承诺跟踪/风险告警/扣分钟/计费/状态机]

M4·M9 回款登记 ──事件──► [③ 承诺桥归因 + 话术库统计(批量)] ──► playbook + pgvector
            │
通话准备 ──► [④ RAG + deepseek-reasoner] ──► 话术建议 → App/Web
物业触发 ──► [⑤ 聚合 + deepseek-reasoner] ──► 手册初稿 → 物业负责人审核 → 版本化
```
> 上述 ①~⑥ 在一期均为 Java 服务内的模块/任务（调云 API 实现），非独立 Python 服务。

- **解析失败是否扣分钟 / 重试策略**（OQ-M5-3）。

## 9. 合规要点（轻量登记）

- **通话录音告知义务**：话术/合同内加录音授权提示。
- **敏感信息脱敏**：发往 DeepSeek / 飞轮语料留存前脱敏（BR-M5-09）。
- **结案可见性收敛**：结案后服务商/催收员只看统计、不看明细（M8 BR-M8-09）——转写/录音/风险记录的可见性受此约束。
- **数据归属**：录音与转写语料归平台（护城河前提），存储在平台自有基础设施。

## 附录 A　录音管线 App 端实现级流程（落地 TD-1）

> 平台：Android（侧载 B2B 应用）。App 只管"拨号→捕获录音→匹配案件→上传→本地暂存"，**不做 ASR / 不算分钟 / 不判风险**（全在服务端）。

### A.1 App 职责边界

| App 负责 | 服务端负责 |
|---|---|
| 选案件→自动拨号、记录通话元数据 | ASR 解析、分钟扣减 |
| 监听并捕获系统录音文件 | 话术抽取、风控、飞轮 |
| 录音↔案件匹配、接通/时长判定 | 转写存储、计费 |
| 上传（队列/重试/离线/去重） | |
| 机型自检上报、送达拍照（另议） | |

### A.2 权限清单

- `CALL_PHONE` — 直接拨号
- `READ_PHONE_STATE` — 通话状态（TelephonyCallback）
- `READ_CALL_LOG` — **通话时长/接通判定主信号**（系统通话记录）
- `MANAGE_EXTERNAL_STORAGE`(Android 11+) / `READ_EXTERNAL_STORAGE` — 读系统录音目录
- `FOREGROUND_SERVICE` + `POST_NOTIFICATIONS`(13+) — 可靠后台上传
- 电池优化豁免引导 — 防国产 ROM 省电杀进程（FileObserver/上传存活关键）

### A.3 核心流程时序（happy path）

```
[选案件] 取号码 → 生成本地 callId, 落库 {callId,caseId,number,dialStartTs}
   ▼
[ACTION_CALL 拨号] App 退后台, 系统通话 UI 接管
   │ TelephonyCallback: OFFHOOK(开始) … IDLE(结束→callEndTs)
   ▼
[通话结束] 并行两路:
   ├─(a) 查 CALL_LOG (number+时间窗) → duration; >0 接通, ≈0/无 → 未接通(不上传)
   └─(b) FileObserver 捕获录音(CLOSE_WRITE/MOVED_TO) → 大小稳定后按 number+[dialStartTs,callEndTs+buffer] 匹配 callId
   ▼
[录音已匹配] 算 fileHash → 入上传队列(Room)
   ▼
[WorkManager 上传] multipart: 文件 + {callId,caseId,dialTs,duration,connected,hash}
   │ 失败→指数退避; 离线→联网续传; 去重→hash
   ▼
[已上传] 服务端解析状态回推(解析中/已解析/失败)
```

### A.4 通话记录状态机（App 端）

```
待拨打 → 拨号中 → 通话中(OFFHOOK) → 通话结束(IDLE)
                                       ├─无录音/duration≈0 → [未接通](终态,不上传)
                                       └─有录音 → 待匹配 → 已匹配 → 待上传 → 上传中 → 已上传
异常: 录音超时未出现 → [录音缺失](告警/重试)   上传失败 → 重试(保留队列)
```
每 callId 一条本地记录，状态持久化（Room），进程被杀/重启可恢复（配合 WorkManager）。

### A.5 三个关键子系统

**A.5a 拨号 + 接通/时长判定（三信号交叉）**
- 拨号：`Intent(ACTION_CALL, "tel:"+number)` 直接拨出，不手输。
- **主信号 = `READ_CALL_LOG`**：通话后查 `CallLog.Calls`（type=OUTGOING、number 匹配、date≈dialTs）取 `duration`；`duration>0` ≈ 接通，duration=通话时长。
- 辅信号 = `TelephonyCallback`(API31+，旧版 `PhoneStateListener`)：OFFHOOK→IDLE 标记起止、触发找录音。注意外呼 OFFHOOK 拨出即触发，**不代表对方接听** → 接听以 CALL_LOG duration + 录音是否存在为准。
- 服务端 VAD 交叉验证有效语音 → 喂 `CFG-MARK-CODES`。

**A.5b 录音文件捕获 + 案件匹配（机制 A 核心）**
- 监听：录音目录起 `FileObserver`，听 `CLOSE_WRITE` 与 `MOVED_TO`（部分 ROM 先写临时文件再重命名）。
- 稳定性：事件后轮询文件大小，连续 N 秒不变再入队（避免半截文件）。
- 匹配：① 文件名解析（多数 ROM 文件名含 `号码_时间戳`，按 ROM 维护规则）+ ② lastModified 落在 `[dialStartTs,callEndTs+buffer]` 且号码归一化一致；取交集，混淆文件名退化为"时间窗+号码"。
- 连续拨打：按时间就近匹配 callId。
- 目录差异（`/storage/emulated/0/Recordings/Call/`、`MIUI/sound_recorder/call_rec/`、华为路径等）由白名单按 ROM 下发；未知机型走候选目录扫描 + 自检引导。

**A.5c 上传队列（可靠性）**
- 本地队列 = Room 表 `{callId,caseId,filePath,fileHash,status,retryCount,createTs}`。
- 上传 = WorkManager（约束联网；可选仅 WiFi）+ 前台服务通知，进程被杀/重启续传。
- 去重 = fileHash（MD5/SHA），服务端再幂等去重。
- 离线留队列，联网自动 flush（对齐"离线暂存+恢复同步"）。
- 上传成功后按策略删本地录音（或保留 X 天）；本地录音加密存储。

### A.6 机型自检 / 白名单流程（启动 + 定期）

```
1. 权限自检 → 缺则引导授权
2. ROM 录音能力探测: 读白名单(本机型ROM的目录/文件名规则/是否已验证); 未命中→引导"测试拨号"检测目标目录是否产出录音
3. 录音可解析性验证: 测试录音能否被 ASR 解(⚠️ 部分华为录音为专有/加密格式, 须验证)
4. 双向音验证: 上传测试录音, 服务端能否分出 2 个说话人
5. 上报: 机型+ROM版本+Android版本+结果 → 后端白名单矩阵增量
6. 结果: 通过→正常用; 不通过→引导开启系统"自动通话录音" / 或提示换推荐机型(落白名单外)
```

### A.7 边界与坑（显性列出）

- ROM 无自动通话录音 → 白名单外，自检拦截（机制 A 固有边界）。
- Android 11+ 分区存储 → 必须 `MANAGE_EXTERNAL_STORAGE`（侧载可用，Play 商店受限）。
- 录音专有格式/加密（华为等）→ 自检第 3 步验证可解析性，不可解落白名单外。
- 后台被杀（国产 ROM 省电）→ 前台服务 + 电池豁免引导 + WorkManager。
- 录音延迟落盘 → 超时轮询窗口 + "录音缺失"告警，不静默丢。
- 双卡/VoLTE/网络电话 → 确认录音是否触发、号码归属；自检覆盖。
- 隐私合规 → 本地录音加密、上传后清理、通话录音告知（BR-M5-09）。

### A.8 与后端接口边界

- `GET 案件号码`（多号码：主号/家属/代接，M4）
- `POST 上传录音`（multipart + 元数据：callId/caseId/dialTs/duration/connected/hash）
- `POST 自检上报`（机型/ROM/结果）
- `GET 解析状态回推`（解析中/已解析/失败 → 驱动 App 状态机末段）
- 服务端据上传触发 ASR、按分钟扣减（TD-2/计费）。

### B.3 承诺桥归因批量 SQL（夜跑/阈值触发，对齐 BR-M5-12 批量增量）

参数：`:win_days`(窗口天数)、`:amt_tol`(金额容差如 0.1)、`:min_n/:min_rate/:min_lift`(入库门槛 = OQ-M5-8 调参)。

```sql
-- ① 承诺↔回款匹配: 每个待履约承诺找"窗口内、金额近、最接近到期日"且未被占用的回款
WITH m AS (
  SELECT c.id AS commitment_id, r.id AS repayment_id,
         (r.paid_at::date - c.due_date) AS day_diff
  FROM commitment c
  JOIN LATERAL (
     SELECT r.id, r.paid_at
     FROM repayment r
     WHERE r.case_id = c.case_id
       AND r.paid_at::date BETWEEN c.due_date - :win_days AND c.due_date + :win_days
       AND r.amount BETWEEN c.amount*(1-:amt_tol) AND c.amount*(1+:amt_tol)
       AND NOT EXISTS (SELECT 1 FROM commitment c2 WHERE c2.matched_repayment_id = r.id)
     ORDER BY abs(r.paid_at::date - c.due_date)
     LIMIT 1
  ) r ON true
  WHERE c.status = '待履约'
)
UPDATE commitment c
   SET status='已兑现', matched_repayment_id=m.repayment_id, fulfilled_days=m.day_diff
  FROM m WHERE c.id=m.commitment_id;

-- ② 回填话术片段结果标签(承诺协商/分期协商场景, 经 source_call_id 串回片段)
UPDATE utterance_snippet s
   SET produced_commitment   = true,
       secured_fulfilled     = (c.status='已兑现'),
       attributed_payment_id = c.matched_repayment_id,
       attributed_amount     = r.amount,
       days_to_pay           = c.fulfilled_days,
       outcome_settled_at    = now()
  FROM call_analysis ca
  JOIN commitment c ON c.source_call_id = ca.call_id
  LEFT JOIN repayment r ON r.id = c.matched_repayment_id
 WHERE s.call_analysis_id = ca.id
   AND s.scene IN ('承诺协商','分期协商');
-- ②b 兜底: 无承诺却直接回款 → 归因到回款前"最后一通接通通话"片段(低权 weak, 统计降权; 略表)

-- ③ 聚合更新平台话术库(按 场景×意图×cohort 上卷)
INSERT INTO playbook_snippet AS p
  (scene,intent_tag,cohort_key,template_text,
   sample_n,commit_cnt,fulfilled_cnt,attributed_pay_cnt,
   commitment_rate,fulfillment_rate,payment_conversion,avg_days_to_pay,last_recomputed_at)
SELECT s.scene, s.intent_tag, s.cohort_key, '(模板待泛化生成)',
   count(*),
   count(*) FILTER (WHERE s.produced_commitment),
   count(*) FILTER (WHERE s.secured_fulfilled),
   count(*) FILTER (WHERE s.attributed_payment_id IS NOT NULL),
   avg(s.produced_commitment::int)::numeric(5,4),
   (count(*) FILTER (WHERE s.secured_fulfilled))::numeric
       / NULLIF(count(*) FILTER (WHERE s.produced_commitment),0),
   avg((s.attributed_payment_id IS NOT NULL)::int)::numeric(5,4),
   avg(s.days_to_pay)::numeric(6,2),
   now()
FROM utterance_snippet s
WHERE s.outcome_settled_at IS NOT NULL
  AND s.scene IS NOT NULL AND s.intent_tag IS NOT NULL AND s.cohort_key IS NOT NULL
GROUP BY s.scene, s.intent_tag, s.cohort_key
ON CONFLICT (scene,intent_tag,cohort_key) DO UPDATE SET
   sample_n=EXCLUDED.sample_n, commit_cnt=EXCLUDED.commit_cnt,
   fulfilled_cnt=EXCLUDED.fulfilled_cnt, attributed_pay_cnt=EXCLUDED.attributed_pay_cnt,
   commitment_rate=EXCLUDED.commitment_rate, fulfillment_rate=EXCLUDED.fulfillment_rate,
   payment_conversion=EXCLUDED.payment_conversion, avg_days_to_pay=EXCLUDED.avg_days_to_pay,
   last_recomputed_at=EXCLUDED.last_recomputed_at;

-- ④ 小样本修正(Wilson 下界, 承诺转化率, z=1.96) + 同cohort基线/lift
UPDATE playbook_snippet p SET
  wilson_lower = (commitment_rate + 1.9208/sample_n
     - 1.96*sqrt((commitment_rate*(1-commitment_rate)+0.9604/sample_n)/sample_n))
     / (1 + 3.8416/sample_n),
  baseline_rate = b.base,
  lift = commitment_rate/NULLIF(b.base,0)
FROM (SELECT scene,cohort_key, avg(commitment_rate) base
      FROM playbook_snippet GROUP BY scene,cohort_key) b
WHERE p.scene=b.scene AND p.cohort_key=b.cohort_key AND p.sample_n>0;

-- ⑤ 入库门槛 → 生效/候选/淘汰(阈值 OQ-M5-8)
UPDATE playbook_snippet SET status = CASE
  WHEN sample_n>=:min_n AND wilson_lower>=:min_rate AND lift>=:min_lift THEN '生效'
  WHEN sample_n>=:min_n AND wilson_lower < :min_rate*0.5                THEN '淘汰'
  ELSE '候选' END;
```

### B.4 设计要点（别踩）

- **承诺桥锚点 = `commitment.source_call_id`**：把"哪通话促成的承诺"显式记下，归因才干净；手动登记承诺无 source_call_id，不进话术归因（只进 M4 跟踪）。
- **防重复计**：`matched_repayment_id` + `NOT EXISTS` 保证一笔回款只归因一个承诺。
- **`template_text` 不在 SQL 里生成**：③只算统计；模板由**单独 LLM 步**对该组高转化片段**泛化 + 剥离 PII**（满足"不吐库本体"），再回写。
- **④ RAG 只读 `status='生效'`**（`idx_pb_lookup` 局部索引），候选/淘汰不进策略建议。
- **节奏**：produced_commitment 抽取时即置（短期）；②~⑤ 批量跑（中期标签晚到）；④策略用库快照——与 BR-M5-12 一致。
- **隐私**：`text_masked` 脱敏、`template_text` 泛化、`source_count` 仅审计不外泄；话术库平台所有（BR-M5-06）。

## 附录 C　④ 策略建议 RAG 检索逻辑（deepseek-reasoner，BR-M5-04）

综合多源（画像 + 本案历史 + 话术库 + 作战手册），两种模式。

**通话前（pre-call）**
```
1. 构造查询: 取案件画像 → 算 cohort_key + 预测可能 scene
   (新案件→首次接触; 多次失联→失联确认; 有争议标记→拒缴争议; 已有承诺→承诺协商/分期)
2. 混合召回 playbook_snippet (status='生效'):
   - 结构化过滤: scene ∈ 预测场景 AND cohort_key 匹配
   - 向量召回: "画像+场景"查询文本 embed → pgvector cosine
   - 融合排序: 0.5*wilson_lower + 0.3*payment_conversion + 0.2*向量相似度
   - cohort 回退: 精确 cohort 样本不足 → 放宽到同 scene 邻近桶, 标 generalized
3. 历史压缩: 本案历史通话多 → 先 map-reduce 摘要(chat 档)再喂 reasoner, 控 token
4. reasoner 组装: system(资深催收教练 + 合规红线:不得教唆辱骂/威胁/虚假承诺)
   + 画像 + 历史摘要 + 检索模板(标"内部参考, 必须改写贴合本案, 不得照抄/不得透露库") + 作战手册要点
5. 结构化输出: {开场建议, 预测异议应对话术N条, 推进目标(拿承诺/约时间/分期台阶),
                风险提示(本案历史风险+合规红线), 待确认信息缺口}
```

**通话后（post-call）**：输入本通 call_analysis + 历史 → reasoner 给下一步策略（承诺跟踪/换号触达/建议法务/二次协商话术/再呼时机）+ **复盘卡片**（做得好/可改进，对接"通话后指导"）。比通话前轻，RAG 可选。

**检索 SQL**
```sql
SELECT id, scene, intent_tag, template_text, wilson_lower, payment_conversion,
       1 - (embedding <=> :q_vec) AS sim
FROM playbook_snippet
WHERE status='生效' AND scene = ANY(:pred_scenes) AND cohort_key = :cohort
ORDER BY (0.5*wilson_lower + 0.3*payment_conversion + 0.2*(1-(embedding <=> :q_vec))) DESC
LIMIT :k;   -- 不足 K → 放宽 cohort_key 到同 scene, 标 generalized
```

**要点**：① 库不外泄靠 prompt 指示改写 + 可加后置高相似度告警（BR-M5-06）；② 用当前库**快照**（BR-M5-12）；③ reasoner 慢 → 案件分配/进工作台时**预热**生成、chat 档兜底；④ 同案画像未变可缓存复用。

## 附录 D　⑤ 作战手册增量 diff 机制（BR-M5-05）

```sql
CREATE TABLE playbook_manual (              -- 作战手册(项目级, 版本化)
  id            bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  project_id    bigint NOT NULL,
  version       int NOT NULL,
  status        text NOT NULL,             -- 草稿/现行/历史
  content       jsonb NOT NULL,            -- 结构化章节: {小区特点,欠费背景,难点,政策,分场景话术[],风险提示}
  generated_from jsonb,                     -- 溯源: {通话量N, 话术库快照ts, 物业录入版本, 上一版id}
  diff_from_version int,
  diff          jsonb,                      -- 相对上一现行版的结构化差异
  created_by bigint, created_at timestamptz DEFAULT now(),
  adopted_by bigint, adopted_at timestamptz
);
```

**触发**（系统提示"作战手册可更新"）：自上次采纳后新增通话量 ≥ 阈值（OQ-M5-4）/ 项目相关话术库条目显著更新 / 物业更新录入情况 / 定期。

**生成流程**
```
1. 输入: 现行版 content(带稳定条目id) + 物业录入(最新) + 项目历史通话分场景聚合 + 高转化话术库条目
2. reasoner 增量修订: "保留有效内容, 仅就新数据更新/补充/修正", 复用现有条目 id, 输出同结构新版
3. 结构化 diff(非文本 diff): 按 (章节key, 条目id) 对齐 → added/removed/modified + 字段级旧/新值 + reason
4. 物业负责人审阅高亮 diff → 逐条/整体采纳或编辑 → 采纳后 version+1 成现行, 旧版转历史
```
diff 示例：
```json
{ "base_version":3, "sections":[
  {"key":"难点","op":"modified","old":"...","new":"..."},
  {"key":"分场景话术","items":[
    {"id":"s_017","op":"added","new":{"scene":"分期协商","text":"..."},"reason":"新数据承诺转化+18%"},
    {"id":"s_004","op":"removed","old":{},"reason":"近90通低于基线, 建议淘汰"}
  ]}
]}
```
**要点**：① **结构化 diff + 稳定条目 id**（否则误判删旧增新）；② **逐条采纳**（被拒 diff 保留旧值，记录采纳/拒绝）；③ AI 不直接发布，**必须人工采纳成现行**，服务商只看现行版（BR-M5-05）；④ generated_from 溯源可复现；⑤ 保留历史版可任意两版对比。

## 附录 E　⑥ 风控规则 + 分级（BR-M5-07，全量 OQ-M5-2 已定）

**风险类型（CFG-RISK-TYPES 初始集建议）**：辱骂/脏话、威胁恐吓、骚扰（频次/时段）、违规承诺（承诺减免/撤诉等无权事项）、冒充（公检法/律师）、隐私泄露（向第三方透露债务）、诱导私人转账、歧视言论。

**两级检测（降本）**
```

 ③ 落库 + 告警 + 处置闭环
```

## 附录 F　call_analysis LLM 抽取 prompt / JSON schema（Stage①，deepseek-chat）

**一次调用同时产出 ①结构化抽取 + ②话术切片 + ⑥语义风险**（省调用）。输入 = 脱敏转写（分句+speaker_id+时间戳）+ 案件画像摘要（供场景判断）+ 注入的 CFG 枚举（scene/risk_type/intent_tag 词表）。

**System prompt 关键条款**
- 你是催收通话分析引擎，**只输出严格符合 schema 的 JSON**，无任何解释文字；`temperature≤0.2`、`response_format=json_object`。
- **仅基于转写，不杜撰**；信息缺失填 null 并降 confidence。
- scene/risk_type/intent_tag **必须从给定枚举选**，无匹配填"其他"。
- **承诺只抽"承诺分期/承诺还款"（催收员通话获得的业主还款承诺），不改应收；与"政策分期（减免规则）"区分**（对齐 BR-M2-16）。
- **金额/日期务必精确**，不确定标 `confidence<0.6` → `needs_human_review=true`（TD-2 红线）。
- 风控只识别记录、**不评判不建议**（建议归 ④）。
- 话术片段**只取催收员侧**（按 speaker_id）。

**输出 JSON schema（示例实例）**
```json
{
  "scene": "承诺协商",
  "debtor_attitude": "抵触",
  "connected": true,
  "dispute_points": ["质疑费用标准"],
  "objections": ["经济困难"],
  "emotion_track": [{"t_ms":0,"speaker":"业主","emotion":"平静","score":0.6}],
  "emotion_delta": 1,
  "commitments": [
    {"amount":2000,"due_date":"2026-07-01","is_installment":false,
     "installment_no":null,"evidence_ts_ms":83400,"confidence":0.9}
  ],
  "risk_hits": [
    {"risk_type":"违规承诺","level":"中","evidence_ts_ms":61000,
     "evidence_text":"我可以帮你免掉滞纳金","rationale":"催收员无权承诺减免"}
  ],
  "next_action": "承诺2000元7月1日前到账, 到期前1天提醒; 违约则升级协调员",
  "snippets": [
    {"intent_tag":"提供分期台阶","speaker":"催收员",
     "ts_start_ms":45000,"ts_end_ms":61000,"text_masked":"这样,我们可以分三期…"}
  ],
  "overall_confidence": 0.85,
  "needs_human_review": false
}
```

**后端落库映射**
- `produced_commitment = commitments 非空`
- `cohort_key = 案件画像(欠费额/账龄,来自M2) + 本输出(attitude/dispute)` 组合计算（LLM 不出 cohort_key）
- `risk_hits → risk_event`；`snippets → utterance_snippet`（再单独 embed 入向量）
- `commitments → commitment 表`（带 source_call_id=本通，作为承诺桥锚点，见附录 B.3）

## 附录 G　风控脏话/模式词表初始集（附录 E 的种子数据，CFG-RISK-TYPES）

| 风险类型 | 规则种子（词表/模式） | 规则可抓 | 默认级别 |
|---|---|---|---|

**维护与反规避（必做）**
- **AC 自动机**（Aho-Corasick）做多词高效匹配；

- **漏报/误报回流**:风控人工复核结果反哺词表/模式迭代；
- 词表/模式/级别全部 **CFG-RISK-TYPES 平台超管可配**。
- 纯语义类(违规承诺/隐私泄露)规则只抓锚点，**主判并入附录 F 的 `risk_hits` 一次 LLM 调用**，不单独再调。

### I.2 存储与更新机制

```sql
CREATE TABLE case_profile (            -- 案件画像物化表(读多写少, 策略建议低延迟)
  case_id          bigint PRIMARY KEY,
```

**更新四类驱动**
