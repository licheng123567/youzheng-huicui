# 前端 1:1 还原 — 后端补字段 / 改契约清单

> 来源:对 PC 全站表单与每角色菜单做「实现端 × 原型 × OpenAPI 契约」三方审计后,梳理出**前端做不了、需后端/契约配合**的差异。前端可独立补的 🟡 项已直接修复,不在此列。
> 优先级:P0=契约漂移(现网读了不存在的字段,有隐患)/ P1=缺端点(导致整页无法 1:1)/ P2=富字段扩展(锦上添花)。

## P0 · 契约漂移(实现已读取、OpenAPI 未定义)
B-04「一次性凭据带外交付」方案 A 未落进契约,前端读取了响应里不存在的字段:

| # | 端点 | 实现读取的字段 | 契约现状 | 建议 |
|---|---|---|---|---|
| 1 | `POST /orgs` (201) | `ownerSetupToken` | 仅返回 `Org{id,type,name,ownerAccountId,status}` | 响应补 `setupToken?: string`(首登一次性令牌) |
| 2 | `POST /members/{id}/reset-password` (200) | `setupToken` | `content?: never`(无响应体) | 补 200 响应体 `{ setupToken?: string }` |
| 3 | `PATCH /orgs/{id}/owner` (200) | `ownerSetupToken` | `content?: never` | 补 200 响应体 `{ setupToken?: string }` |
| 4 | `POST /risks/{id}/escalate` | 实现发 `{ note }` | `requestBody?: never` | 补 `EscalateInput { note?: string }`(与 dispose/review 一致) |

> 不补则:前端要么改取数路径(凭据无法展示),要么 escalate 的 note 被后端忽略。建议优先补 #1–#3(凭据是上线流程必需)。

你已决定"都建",但以下三页**当前没有数据源**,需后端先加端点,前端才能建成真页(否则只能空壳):

| # | 页面(角色) | 需要的端点 | 说明 |
|---|---|---|---|
| 5 | 法务 legal(PC) | `GET /legal-docs?scope&status&page` | 现仅 `/cases/{id}/legal-docs`(按案件)。需跨案件法务文书列表(本范围裁剪) |
| 6 | 我的缴费链接 myLinks(PC/CO) | `GET /pay-links?mine&status&page` | 现仅 `POST /cases/{id}/pay-links`。需"我发起的/本范围"缴费链接列表 + 重发/作废入口 |

## P2 · 富字段扩展(原型有、后端 v1.0 未纳入;要 1:1 富表单才需要)

| # | 模型/端点 | 原型字段(后端缺) | 影响 |
