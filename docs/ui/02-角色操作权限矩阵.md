# 有证慧催 UI · 02 角色操作权限矩阵

> 依赖：`00-设计总纲.md` §6（一页多角色复用机制）、`docs/prd/M1` 权限、`M8` 结案收敛、§9 隔离。
> 用途：把"同一页面、各角色操作不同"逐条落成可实现的权限/脱敏规则。这是高保真原型中**角色切换器**的行为依据。
> **本矩阵是 UI 显隐口径，最终权限以后端权限点字典为准；负责人可在其权限子集内再裁剪员工（BR-M1-03）。**

## 3. 案件详情页（P-DATA-09）— 复用核心

> 同一套布局，所有 PC 角色进入看到**相同信息架构**；下表为差异。**SE 列权限以各单元格为准**：默认仅 `case.assign`/`case.dispatch`（派单/分配）写权限，余项随 SA 但**多为只读或无入口**，并一律叠加**数据范围授权（▲）**；切勿据本导语推断 SE 拥有 SA 全部写权限。

## 9. 跨切面脱敏与隔离规则汇总（UI 必须内建）

| 规则 | 触发 | UI 表现 |
|---|---|---|
| 结案收敛（BR-M8-09） | 案件结清/撤案/坏账 | VL/CO 进入案件 → 仅"结案统计视图"，明细标签页/操作全部消失 |
| 跨服务商隔离（BR-M1-05） | 非本商案件（除开放抢单） | 列表过滤不出现；直链返回无权限、不泄露存在 |
| 资金隔离（§2 强约束） | 物业看付佣 / 服务商看收佣 | 无菜单、无字段、无接口入口 |
| 平台员工数据范围（BR-M1-14） | 区域/服务商/物业范围外对象 | 列表过滤、详情拒访、报表只算范围内 |
| 催收依据可读、佣金不可读（BR-M1-06） | 服务商看项目档案 | 催收依据区块只读可见；佣金/对账区块不渲染 |
| 代操作透明（BR-M1-15） | 平台代物业/服务商操作 | 成功 toast"已记录操作日志"；被代方在操作日志可见该条 |
| 权限缺失不渲染 | 无权限点 | 按钮/菜单/标签页不出现（非灰显） |

## 10. 原型实现映射（指导后续若依开发）

```js
// 原型用：当前角色 → 权限点集合 + 脱敏策略
const ROLE = {
  // 以下权限码已对齐契约 x-permission（02-角色操作权限矩阵 + openapi-core.yaml 双源核对）
  // 历史伪码→契约实际码：case.mark→case.follow, case.link→case.paylink,
  //   case.repay→case.repay.mark, case.legal.start→legal.create, case.evidence→evidence.create
  CO: { perms: ['case.call','case.follow','case.promise','case.paylink','case.release'],
        scope: 'ownProviderSeaAndPrivate',
        redactWhenClosed: true },
  PC: { perms: ['case.call','case.repay.mark','legal.create','evidence.create','case.close','case.paylink'],
        scope: 'ownProperty', redactWhenClosed: false },
  // ... SA/SE/PL/VL
}
// 渲染：按钮 v-if="hasPerm(currentRole,'case.close')"；字段 v-if="canSee(field,currentRole,case)"
```

> 一套页面组件 + 一份权限点字典 + 一份脱敏策略 = "看到的一样、能操作的不一样"。这正是用户强调的复用本质。
