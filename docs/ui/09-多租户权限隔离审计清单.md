# 多租户SaaS 分角色权限/数据隔离审计清单

> 针对多租户SaaS系统的RBAC越权、菜单不一致、数据范围泄漏等常见审计痛点，提供 **8~12条可操作核对项** 指导角色一致性审计。基于有证慧催项目实际演进经验（6轮审计/对抗校验）总结易错点与检查路径。

---

## 核心审计框架

**两大维度**：
1. **权限边界审计** — 识别 RBAC 越权、菜单显隐不一致
2. **数据隔离审计** — 排查多租户数据泄漏、脱敏不彻底

---

## 清单：8条高频易错项（按审计优先级）

### 1. ✓ **菜单可见 ≠ 功能可操作（导航层隐藏/功能层未强制）**

**问题描述**：导航菜单中隐藏了某菜单项，但直链访问或通过 API 调用仍可越权操作（"nav 可见但动作未强制"）。

**检查路径**：
- [ ] **导航字典与权限点一致性**：遍历所有菜单项，确认 `navMeta` 的 `roles` 白名单与后端 RBAC 权限点一致（勿单靠前端 nav 过滤）
  - 例：物业协调员(PC)的菜单无"成员管理"，但直链 `/settings/members?org=xxx` 是否能访问？
  - 审计代码行：`if(nav.roles.includes(role)) { render() } else { redirect }` 需后端二次校验
- [ ] **按钮操作与权限守卫一致**：不仅前端 `v-if="can('permission.key')"` 隐藏按钮，还要后端 API 调用时强制权限校验
  - 反例：CO 标注跟进按钮隐藏，但 POST /case/mark 未校验 caller 是否 CO
  - 取证点：grep 后端路由层 `@Permission('case.mark')` 装饰器是否全覆盖

**常见遗漏**：
- 导出/下载/打印操作无权限点校验（PL 本不该导出全平台报表，但 GET /report/export 无 org 范围检查）
- 高危操作(删除/转移/作废)仅前端隐藏，后端无二次确认对话 + 权限叠加

---

### 2. ✓ **数据可见但操作权限未隔离（只读权限溢出）**

**问题描述**：用户能看到某行数据(◐ 脱敏可见)，但表格的"编辑/删除/转移"按钮未被正确隐藏，导致权限越界。

**检查路径**：
- [ ] **字段级权限与操作权限的二重检查**：
  - 例：PL 可见案件的"风险标签"(◐ 脱敏)，但"一键标记风险"按钮是否已隐藏？
  - 审计代码：数据列表遍历行时，`v-for="item in items"` 内的按钮需叠加 `v-if="can('action.key') && isAllowed(item.property)"`
  - 否则：只要数据返回前端，任何按钮点击都可能触发越权API
  
- [ ] **表格/卡片列表的"批量操作"防护**：
  - 批量勾选后"删除/变更/转移"按钮是否校验每项的权限？
  - 反例：勾选多个案件后"批量派单"，后端仅校验了 caller 有 `case.dispatch` 权限，未逐项验证派单目标对象的归属

**常见遗漏**：
- 质检看板中 VL(服务商负责人) 可见本商案件的高风险标记(◐)，但"转质检/通知"按钮仍可点击
- 对账单页面中 PC(协调员) 可见 PL 的对账确认状态(只读)，但"驳回/作废对账"按钮未隐藏

---

### 3. ✓ **多租户数据范围未完全过滤（跨租户泄漏）**

**问题描述**：同一查询在不同角色身份下返回的数据集合应不同，但代码中硬编码过滤条件或缺少 `WHERE org_id=?` 之类的范围约束。

**检查路径**：
- [ ] **数据查询的租户字段强制过滤**：
  - 每一条查询语句前后端都应包含租户/归属检查：
    ```sql
    -- 不安全：SELECT * FROM cases WHERE status='进行中'
    -- 安全：SELECT * FROM cases WHERE status='进行中' AND property_id=${propertyId}
    ```
  - SE(平台员工)查询案件时需加 `WHERE property_id IN (SELECT id FROM properties WHERE dataRange CONTAINS ${userId})`
  - 取证：grep 所有 SELECT/UPDATE/DELETE 语句，确认 `WHERE` 子句包含租户字段

- [ ] **列表聚合与统计字段的数据范围隐藏**：
  - 例：报表中"全平台回款率"应对 SE 过滤为"授权范围内回款率"，对 CO 过滤为"本商案件"
  - 反例：前端显示过滤正确，但后端 API `/api/stats/recovery-rate` 若无 `?dataRange=myScope` 参数传递，即使前端传了也可能被忽略
  - 审计路径：trace API 调用链，确认数据范围参数从前端→中间件→DB 全程传递

- [ ] **跨租户关系表的"可见父对象=可见子对象"假设**：
  - 例：PL 可见本物业的批次，应自动仅见本物业批次内的案件（不需二次授权）
  - 但若代码分离查询（先查 batch，再查 case），中间可能遗漏范围校验
  - 审计：逐个三层关系(property → batch → case)依次grep过滤条件

**常见遗漏**：
- SE 质检看板按钮"转质检"后台无 dataRange 过滤，导致可将不授权范围内的案件转质检
- 充值/计费明细查询前端过滤了"本物业/本商"，但后端 API 若返回所有数据，前端组件卸载时仍暴露原始数据
- VL 查看付佣对账时应看不到 PL 的收佣对账单据，但数据库若无硬隔离(如物理表或 view)，join 时易出错

### 4. ✓ **脱敏规则遗漏（敏感字段/结案后未脱敏）**

**问题描述**：系统定义了某些敏感字段(如业主联系方式、律师函内容、减免信息)或结案后的完整明细，应在特定角色或状态下脱敏，但实际渲染时未过滤。

**检查路径**：
- [ ] **列表态的敏感字段脱敏**：
  - 案件列表中业主手机/身份证号应在结案后对 VL/CO 脱敏（但对 SA/PL 仍可见）
  - 审计：选择一个结案案件，切换 VL/CO 角色后刷新列表，业主联系方式列应显示 `****`
  - 代码检查：`field(item.phone)` 函数内需判断 `if(isClosedCase && redactClosed.includes(role)) return mask(phone)`

- [ ] **详情页的完整明细脱敏**：
  - 结案后 VL/CO 进入案件详情，应直接被重定向到"结案统计视图"(仅计数/转化率)而非加载原详情页
  - 反例：跳转链接仅前端路由控制，用户修改URL直链 `/case/12345` 仍能加载全字段
  - 审计路径：作为 VL，尝试在结案案件上点击"查看详情"，确认是否进入专用统计页而非标准详情页

- [ ] **子数据(工单/承诺/通话记录)的层级脱敏**：
  - VL/CO 对结案案件的"跟进记录"、"通话转写"、"承诺详情"应全部隐藏(显示 `—`)而非脱敏显示
  - 审计：case.callRecords、case.notes、case.promises 是否都检查了 `if(canSeeDetail(case)) render(child) else return`

**常见遗漏**：
- 法务草稿/律师函模板中的金额计算规则对 CO 泄漏(应脱敏)
- H5 业主缴费页面的"催收详情"对过期 token 用户仍部分可见(XSS 风险)
- App CO 打完电话标注后，结案前后的通话记录权限切换点错误

---

### 5. ✓ **批量操作的权限聚合校验缺失（同⊄后端无校验）**

**问题描述**：用户在前端批量勾选 N 条数据，后端在处理批量请求时仅校验了一次权限(caller 拥有 permission.key)，但未逐项验证每条数据是否真的被该用户拥有或授权。

**检查路径**：
- [ ] **批量请求时的"逐项校验"**：
  - API 签名：`POST /case/batch-dispatch { caseIds: [...], targetVL: 'xxx' }`
  - 不安全：后端仅 `if(user.can('case.dispatch')) { dispatch(caseIds) }`
  - 安全：`for each caseId { validate(caseId.property IN user.scope); validate(caseId.status === '待派单'); dispatch() }`
  - 取证：grep 批量接口，查 for-loop 或 bulk update 语句内是否每项都加权限/数据范围检查

- [ ] **客户端侧勾选框的权限限定**：
  - 是否只允许勾选"当前用户有写权限的"行？
  - 例：PC(协调员) 在案件列表只能勾选"本物业的进行中案件"做"登记承诺"，不能勾选"待派单"或"其他物业"的行
  - 审计代码：`item.disabled = !(item.property === myProperty && item.status === '进行中' && can('case.promise'))`

**常见遗漏**：
- 批量派单时后端未校验 `targetVL` 对应服务商是否被授权给这批批次
- 批量标记风险时未过滤"非 SA/SE 无法批量标记"这一限制(前端隐藏了按钮，但API无校验)
- 批量下载对账单时对 SE 无数据范围过滤(导出了不授权的物业数据)

### 6. ✓ **角色切换器行为与实际权限不同步（代理/委托操作无痕迹）**

**问题描述**：系统支持高权限角色(SA)代理低权限角色(PL)执行操作，但代理关系、被代人身份、操作时间等信息未被正确记录在审计日志中，或前端切换角色后权限缓存未刷新。

**检查路径**：
- [ ] **代理操作的审计日志**：
  - SA 代 PL 做"撤案"或"冲正回款"操作时，操作日志应记录：
    ```
    operator: SA-Alice
    operation: 案件撤案
    target: Case#12345
    onBehalfOf: PL-Bob(物业A)  ← 被代物业信息
    timestamp: 2026-06-19 10:23:45
    ```
  - 审计代码：`logOp()` 函数的第 3 参 `proxyFor/targetOrg` 是否被传递并写入 DB
  - 反例：有证慧催08报告中 **[A项] doClose 未传 proxyFor** — SA 代结案的审计日志完全看不出"谁被代结案了"

- [ ] **角色切换后的权限缓存刷新**：
  - 前端"角色切换器"变更 role 后，导航菜单、操作权限、数据范围应立即生效
  - 反例：页面状态(如选中案件、表单数据)未被重置，切换角色后仍沿用旧权限查询、导致跨租户泄漏
  - 测试路径：
    1. 以 PL-A(物业A) 登录，打开案件详情
    2. 通过角色切换器切到 PL-B(物业B)
    3. 观察当前案件是否被重置(应无法访问/自动跳转)，且列表应刷新显示 B 的案件

- [ ] **"代理模式下的权限范围"是否收窄**：
  - SA 代 PL 操作时，SA 应被限制在"仅操作该 PL 可见的数据"而非拥有 SA 的全量权限
  - 审计：SA 登录后选择"代理 PL-A"，尝试派单一个不属于 PL-A 的批次，应被拒绝

**常见遗漏**：
- 审计日志中记录了操作但没有 proxyFor 字段，被代人查看操作历史时无法识别谁代自己操作的
- 前端 `localStorage` 缓存角色后，角色切换时仅改变 `role` 变量，但 `perms`/`dataRange` 缓存未清除
- SA 代理 VL 时，VL 的"仅见本商"限制被绕过，导致看到其他服务商数据

### 7. ✓ **高危操作无二次确认或受控原因选项缺失**

**问题描述**：撤案、坏账结案、冲正回款、停用账户等高危操作应弹出二次确认对话，且某些操作需用户从预定义的"原因"列表中选择(而非自由文本)，否则可能导致误操作或审计追溯困难。

**检查路径**：
- [ ] **高危操作清单与二次确认覆盖**：
  - 遍历系统中所有"删除/转移/状态反向"操作：
    - 撤案/坏账结案：是否弹 Modal 显示"确认要撤案吗？原因是？"
    - 冲正回款：是否弹 Modal 显示"确认冲正？原因？操作不可逆"
    - 案件作废(待派单)：是否弹 Modal + 选择原因下拉
  - 审计代码：grep `confirmDialog / askConfirm / Modal.warn`，确认所有危险操作都有对应 caller
  - 反例：有证慧催08报告中 **[C项]** 批次结清触发出单无操作入口；**虽不属二次确认，但属"受控流程缺失"**

- [ ] **结果单选/原因下拉的受控枚举**：
  - 结案时"原因"应从 `CFG-CLOSE-REASONS` 枚举中单选，而非任意文本输入
  - 释放回公海时"原因"应从 `CFG-RELEASE-REASONS` 中选择
  - 审计：表单中相关字段是否为 `<select options={cfgs.CLOSE_REASONS}>` 而非 `<input type="text">`
  - 取证点：`index.html L2036-2051` 结案弹窗内应有 `<select>` 且禁用提交直到选择了原因

- [ ] **二次确认对话的权限边界**：
  - 是否所有角色的高危操作都有二次确认？还是仅 SA 需要确认而 PL 直接执行？
  - 根据 PRD，应统一规则(建议所有角色二次确认)而非差异化

**常见遗漏**：
- "作废"操作直接执行，未弹确认窗，导致误点后无法撤销
- 结案时虽然弹了确认窗，但"原因"仍为自由文本而非下拉(审计难以分类和统计)
- 冲正回款时未提示"不可逆"且未要求输入原因和操作人确认

### 8. ✓ **跨端隔离不一致（PC/App/H5 权限口径不同步）**

**问题描述**：多端应用(PC Web + Mobile App + H5 业主页)中，同一操作在不同端的权限控制不一致。例如 App 上 VL 能看到某字段，PC 上被隐藏，或 H5 缺少必要的令牌校验导致无登录用户也能访问账单。

**检查路径**：
- [ ] **App 端的结案脱敏与 PC 一致性**：
  - CO 在 App 中打完电话后结案，与 PC 网页中的脱敏规则应一致
  - 反例：有证慧催06报告指出 **App 结案未脱敏**(当时认为是隐患)
  - 测试路径：
    1. CO 在 App 中浏览"进行中"案件详情(应见通话记录)
    2. 标注为"已结清"后，继续访问该案件(应被脱敏转跳)
    3. 在 PC 网页以同一 CO 身份验证脱敏规则是否一致

- [ ] **H5 业主页的令牌门控(token-based access)**：
  - H5 不涉及登录，而是通过"查看账单"链接中的令牌(`?token=xxx`)来校验身份
  - 问题：token 是否在账单支付/批次撤案后失效？
  - 审计代码：`h5.html` 中 IIFE 是否检查 token 白名单 + 令牌过期时间
  - 取证：`h5.html L445` 应检查 `TOKENS.includes(token) && !isExpired(token)` 而非所有 token 都有效

- [ ] **App-PC 菜单可见性同步**：
  - PL/PC 在 App 上的导航菜单与 PC 网页应相同(或有文档说明差异)
  - 反例：PC 端有"成员管理"菜单但 App 无，用户可能困惑权限是否正常

**常见遗漏**：
- App CO 的工作台(今日待办)与 PC 私海列表的数据范围应一致，但如果 App 缺少 dataRange 参数传递会导致跨租户
- H5 在用户改手机或注销后，旧 token 仍可访问账单(token 失效机制缺失)
- PC Web 中 PL 的某操作权限已禁用(前端 disabled)，但 App 仍可点击(权限点未同步)

## 进阶审计项（4条补充）

### 9. ✓ **权限点与操作的"关键路径"映射验证**

**问题描述**：后端权限点字典定义了权限，但前端的具体操作与权限点的绑定是否准确？例如"登记承诺"操作绑定的权限点是否真的是 `case.promise`？

**检查路径**：
- [ ] **权限点全覆盖清单**：
  - 建立一份"权限点字典"(如 `case.dispatch`, `case.mark`, `case.evidence` 等)
  - 针对每个权限点，grep 代码确认：
    - (1) 后端是否有对应的权限校验装饰器 `@Perm('case.promise')`
    - (2) 前端是否有对应的 `can('case.promise')` 检查
    - (3) 是否存在"权限点定义了但从未被检查"的孤岛点
  - 反例：定义了 `case.link`(关联工单)权限，但前端检查了 `can('case.mark')`(标注)，两者混淆

- [ ] **权限点的"最小化原则"验证**：
  - 每个角色应拥有最少必需权限；是否存在过度授权？
  - SE 应仅有 `['case.dispatch', 'case.assign', 'report.export']` 三项，其他操作应无权限
  - 取证：查后端 role_permissions 表或 RBAC 配置文件

---

### 10. ✓ **API 请求参数中的"隐藏权限"校验(竞态条件)**

**问题描述**：某些 API 请求虽然在权限层面通过了认证，但请求参数本身可能包含用户无权操作的目标。例如派单时指定了不属于任何授权服务商的 targetVL，或查询时指定了不在数据范围内的 property_id。

**检查路径**：
- [ ] **请求参数的"目标对象校验"**：
  - API：`POST /case/dispatch { caseIds: [...], targetVL: 'xxx' }`
  - 校验：后端需确认 `targetVL` 要么属于 caller 自己(VL→VL 分配)，要么属于平台授权的列表(SA 指派给任意 VL)
  - 不安全：`if(user.can('case.dispatch')) { dispatch(targetVL) }`
  - 安全：`if(user.can('case.dispatch') && (user.role === 'SA' || targetVL === user.vlId)) { dispatch() }`

- [ ] **关系型查询中的归属验证**：
  - 查询 `/api/batches/{batchId}/cases`，是否校验了 batchId 属于 caller 授权的物业/服务商？
  - 审计路径：所有涉及 ID 查询的 API 都应有"先查关系，再验证归属，再返回"的三步流程

---

### 11. ✓ **数据脱敏的"演示数据暴露"问题(演示环境的权限假象)**

**问题描述**：系统使用 mock 数据做演示时，权限检查逻辑可能因为数据同租户而"看不出差异"。例如 SE 和 VL 虽然权限不同，但 demo 数据都属于"阳光花园"物业，不足以验证跨租户隔离。

**检查路径**：
- [ ] **演示数据的"多租户代表性"**：
  - mock 数据应包含至少 3 个租户(3个物业/3个服务商)，确保权限过滤、隐藏能在演示中可见
  - 例：SE 本应仅见"阳光花园"批次，但若 demo 只有这一个物业，无法验证 SE 是否正确过滤了其他物业
  - 取证点：`index.html L2500+` 的 `allBatches` 应包含多个物业，SE 的 `scopedBatches` 应是子集

- [ ] **权限缺陷的"演示可能性"文档**：
  - 在审计报告中标注"此缺陷在演示环境中因数据同租户而不可见，但逻辑真实存在"(见有证慧催08报告中"低 severity 演示弱"项)
  - 这有助于开发阶段优先级排序

---

### 12. ✓ **三层隔离规则的"综合验证"(整体审计而非零散检查)**

**问题描述**：前述各项审计可能出现"局部正确整体错误"的情况。例如菜单隐藏正确、数据过滤正确、但脱敏规则与操作权限冲突，导致综合审计失败。

**检查路径**：
- [ ] **"用户故事"全链路走查**：
  - 选定一个关键操作流程，以不同角色身份全程走查，记录每个环节的权限/数据表现
  - 案例：**催收员结案全流程(CO视角)**
    1. 登录 → 工作台 TODO(应见"今日待标注案件")
    2. 进案件列表 → 筛选进行中 → 点击进入案件(应见通话/标注区)
    3. 标注结果 → 填承诺 → 点"结案"→ 弹二次确认窗 + 原因下拉 → 结案
    4. 返回列表 → 案件从"进行中"变"已结清" → 点该案件 → **应被重定向到脱敏视图**而非详情页
    5. 查看操作日志 → 应见"结案"记录但不见其他物业案件的操作
  - 任何环节失败(如第 4 步能进详情页)则整体审计失败

- [ ] **"权限矩阵↔UI原型↔审计报告"三角验证**：
  - 权限矩阵定义了 CO 对"案件详情"的可见性为 `●(编辑标注)`
  - UI 原型中 CO 确实能点标注按钮
  - 但审计报告若发现结案后该按钮仍可点，则为真实缺陷
  - 结论：三者应形成"一致性三角"，任何两个一致第三个偏离则认定为缺陷

**常见遗漏**：
- 单独检查"菜单"通过，单独检查"数据过滤"通过，但综合流程中发现中间步骤有权限漏洞
- 权限矩阵中 ▲ 符号表示"受数据范围限制"，但实际代码中仅做了角色检查未做范围检查

## 总结

多租户 SaaS 权限隔离审计的核心是**三层防线**：
1. **导航与权限一致**(第 1 项) — 防"菜单可见但无权操作"
2. **数据可见与写操作隔离**(第 2、3 项) — 防"只读字段仍可编辑"
3. **高危操作的受控流程**(第 4、6 项) — 防"误操作与操作追溯"

建议建立**角色一致性矩阵**(类似有证慧催 02 文件)，逐次审计 12 项检查表，最终通过第 10 项的全流程走查来综合验证。
