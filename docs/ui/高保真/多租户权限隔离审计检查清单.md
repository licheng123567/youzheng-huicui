# 多租户SaaS 分角色权限/数据隔离审计检查清单

## 核心原则
多租户隔离失效的根本原因：菜单可见性控制 ≠ 后端数据权限检查。审计需要**穿透UI层**，在业务逻辑和数据库层面验证权限边界。

---

## 8-12条可操作核对项

### 一、RBAC越权检查（3项）

**1. 菜单可见性 vs 实际API权限一致性**
- [ ] 检查清单：
  - 前端隐藏某功能菜单后，直接调用对应API是否被后端拒绝
  - 测试方法：前端置灰/隐藏按钮，用浏览器开发工具或Postman直接请求该端点，验证返回403/401而非200
  - 易错点：前端只做了菜单控制，后端API未检查当前用户/角色权限
  - 审计示例：财务角色看不到"删除订单"菜单，但直接POST /api/order/delete/{id}仍成功

**2. 同一用户在不同租户中的权限隔离**
- [ ] 检查清单：
  - 同一账号在租户A是"普通员工"，在租户B是"管理员"，数据访问权限是否严格按租户+角色隔离
  - 测试方法：用租户B的admin token尝试访问租户A的数据，应返回无权限错误
  - 易错点：认证通过后仅记录user_id，未在每次请求时校验tenant_id是否属于当前用户
  - 审计示例：登出再登录其他租户后，旧的tenant_id是否仍能读取数据

**3. 权限继承链的完整性（特别是间接权限）**
- [ ] 检查清单：
  - 用户-角色-权限三层关系中，删除用户后对应权限是否自动清理
  - 修改角色权限后，已分配该角色的用户是否立即生效（而非缓存导致延迟）
  - 测试方法：修改角色权限 → 用该角色用户再发请求 → 验证新权限生效；或设置过期token缓存，观察权限更新延迟
  - 易错点：权限缓存未失效、删除操作未级联、动态权限判断逻辑错误
  - 审计示例：移除用户的某权限后，用户仍能短时间内使用该功能

---

### 二、导航与动作权限不一致（2项）

**4. 详情页面/编辑页入口检查**
- [ ] 检查清单：
  - 某个详情/编辑页面虽然在导航中被隐藏，但用户知道URL后直接访问是否被拦截
  - 测试方法：扫描前端路由表，对每个受权菜单对应的路径，绕过导航直接访问
  - 易错点：前端路由只在根据权限条件渲染菜单项，但Router.beforeEach未检查权限；或只在页面mount时检查，但异步调用数据时未检查
  - 审计示例：打开/hr/salary-list被拦截，但/hr/salary-detail/123直接访问成功

**5. 批量操作权限检查（导出、批删、群发等）**
- [ ] 检查清单：
  - 批量导出数据是否对每条数据逐一检查权限，而非仅检查表单入口
  - 批量删除/更新是否在后端逐条验证当前用户是否有权操作每条记录
  - 测试方法：导出表格时手工删改CSV中的record_id，再上传验证是否能修改自己无权的记录
  - 易错点：前端勾选了3条记录，后端直接WHERE id IN (...)而未检查租户/行权限
  - 审计示例：普通员工导出整公司的工资表（虽然前端筛选了自己的数据，但API参数可改）

---

### 三、数据范围与隔离检查（2项）

**6. 列表查询的租户/部门过滤**
- [ ] 检查清单：
  - 所有列表接口（GET /api/users, /api/orders等）是否在WHERE子句中自动加上租户筛选条件
  - 测试方法：抓取请求 → 修改查询参数（如?tenant_id=otherTenant或?department_id=）→ 验证后端拒绝或忽略该参数
  - 易错点：前端UI只显示当前租户数据，但API未加tenant_id过滤，改参数就能看其他租户
  - 审计示例：
    ```
    GET /api/orders?status=pending
    # 应自动变成
    WHERE status='pending' AND tenant_id=current_tenant AND (created_by=me OR role_has_permission)
    # 不应被?tenant_id=hack覆盖
    ```

**7. 关联数据的跨租户泄漏检查**
- [ ] 检查清单：
  - 用户详情页能否看到不属于当前租户的用户信息（通过直接修改URL中的user_id）
  - API返回的嵌套关联对象（如订单中的客户信息、员工中的部门信息）是否来自其他租户
  - 测试方法：在A租户中获取一个租户B用户的ID → 直接访问/api/users/{b_user_id} → 应返回404或403
  - 易错点：查询时未检查关联对象的tenant_id、使用了user_id作为唯一查询条件（无租户前缀）
  - 审计示例：GET /api/user/5 返回了租户B中user_id=5的信息，而当前登录用户属于租户A

---

### 四、数据脱敏与敏感信息泄漏（2项）

**8. 密码/令牌/API密钥在API响应中的脱敏**
- [ ] 检查清单：
  - 用户列表/详情接口是否返回password_hash、api_token、secret_key等敏感字段
  - 编辑表单提交时，服务器响应是否回显用户输入的密码（即使是*号也不该显示）
  - 测试方法：
    - 调用GET /api/users → 查看返回JSON是否包含password、token、secret等字段
    - 修改用户信息后的response是否包含敏感信息
  - 易错点：在DTO/序列化层未排除敏感字段、直接返回了数据库model对象
  - 审计示例：GET /api/users/123返回了 `{"id":123, "name":"Alice", "password":"$2b$12...", "apiToken":"sk_live_xxx"}`

**9. 行级数据的脱敏（财务数据、健康数据、个人信息等）**
- [ ] 检查清单：
  - 报表/统计中的个人敏感数据（薪资、医疗记录、身份证号）是否对非授权角色脱敏
  - 导出数据时，用户是否能选择导出所有字段或仅导出自己可见的字段
  - 测试方法：以普通员工角色导出报表 → 验证是否包含所有人的薪资、是否只显示自己的数据
  - 易错点：前端隐藏敏感列，但导出时后端未过滤；或导出使用的是不同的权限检查逻辑
  - 审计示例：员工看报表时薪资被*号脱敏，但导出Excel后薪资完整可见

---

### 五、角色一致性与权限模型检查（2-3项）

**10. 权限模型的完整性（不同角色的权限组合逻辑）**
- [ ] 检查清单：
  - 系统中所有角色的权限定义是否在代码/文档中有明确矩阵表
  - 是否存在"应该互斥但未互斥"的权限组合（如同时拥有"查看所有员工"和"仅查看下属"）
  - 测试方法：列举所有角色 → 为每个角色分配权限 → 验证是否有逻辑矛盾
  - 易错点：权限矩阵和实际代码不同步、新增功能时未更新角色权限
  - 审计示例：
    ```
    角色权限矩阵
    | 权限 | 普通员工 | 部门主管 | HR | 财务 | 管理员 |
    |------|---------|--------|-----|------|--------|
    | 查看工资 | 自己 | 下属 | 全部 | 全部 | 全部 |
    | 修改工资 | ✗ | ✗ | ✓ | ✓ | ✓ |
    | 删除工资 | ✗ | ✗ | ✗ | ✗ | ✓ |
    ```

**11. 动态权限策略的一致性（属性/资源级别权限）**
- [ ] 检查清单：
  - 如果系统支持"自定义角色"，新建角色时是否有权限清单的审核/预设限制
  - 同一操作在不同上下文中的权限是否一致（如"部门主管可修改下属的xx字段"这条规则是否全局应用）
  - 测试方法：创建自定义角色，尝试赋予不合理权限组合 → 后端是否接受或拒绝
  - 易错点：权限检查逻辑分散在不同模块，导致同一操作在A界面被允许、B界面被拒绝
  - 审计示例：批量修改员工时"修改部门字段"被允许，但个别修改时被拒绝

**12. 权限变更日志与审计追踪**
- [ ] 检查清单：
  - 所有权限相关的变更（用户角色变更、权限新增删除、特殊权限授予）是否有不可篡改的操作日志
  - 日志中是否包含操作人、操作对象、操作前后对比、时间戳、IP地址等信息
  - 测试方法：修改某用户的角色 → 查询audit_log表 → 验证日志记录完整
  - 易错点：日志存在但可被同权限用户修改、日志字段不完整、关键操作无日志
  - 审计示例：给普通员工授予管理员权限 → 应在audit_log中记录"who, what, when, why"

## 审计执行流程

### 第一阶段：文档与配置审查
1. 获取角色权限矩阵文档（RBAC模型定义）
2. 获取数据隔离策略文档（多租户隔离规则）
3. 交叉验证代码与文档的一致性

### 第二阶段：静态代码审查
- 搜索关键字：`checkPermission`, `authorize`, `tenant_id`, `WHERE`, `filter`
- 检查每个API端点是否都有权限检查中间件
- 检查DTO/Response序列化是否排除敏感字段

### 第三阶段：动态测试（黑盒）
- 以不同角色登录，测试上述12条核对项
- 使用Burp Suite/Postman修改请求参数（tenant_id, user_id, role等）
- 验证后端的拒绝/接受响应

### 第四阶段：日志审计
- 检查权限相关操作的日志完整性
- 尝试从日志追踪某个越权事件的完整链路

## 常见易错模式与修复建议

| 易错模式 | 表现 | 修复建议 |
|---------|------|---------|
| 前端权限控制作为唯一防线 | 隐藏菜单但API仍可调用 | 每个API端点都需要后端权限检查 |
| 忘记加租户过滤 | 查询参数可被改成其他租户ID | 使用数据库视图/中间件自动注入tenant_id |
| 密钥/密码未脱敏 | 导出或API响应中暴露 | 在DTO序列化时@JsonIgnore或@Exclude敏感字段 |
| 权限缓存不失效 | 修改角色后用户权限未立即更新 | Redis缓存加TTL或使用权限变更事件触发清缓存 |
| 批量操作未逐条检查 | 导出/批删时权限控制失效 | 在循环中检查每条记录的权限，而非整体检查 |
| 关联查询跨越租户边界 | JOIN查询未加租户过滤 | SQL中显式加`AND tenant_id = ?` |
| 日志可被删除或篡改 | 无法追踪越权事件 | 使用append-only日志库或数据库触发器 |
