# 多租户SaaS分角色权限/数据隔离审计 — 核心12条检查清单

> 可操作的要点列表，用于指导多租户系统的角色一致性审计。基于有证慧催6轮审计的实战总结。

---

## 📋 核心12条检查项 (按审计优先级)

### 🔴 HIGH优先级（必须通过 ≥11/12）

#### 1. **菜单隐藏 ≠ 功能禁用 (前端导航vs后端权限不一致)**

**快速诊断**
- [ ] 前端菜单隐藏了"成员管理"，直链 `/settings/members` 是否能访问？
- [ ] 按钮隐藏了(v-if="false")，调用 API 是否仍成功？
- [ ] 导出/下载/打印操作是否都有后端权限装饰器检查？

**修复要点**
- 所有路由都加 `@Permission('perm.key')` 装饰器
- API 校验逻辑：`if(!user.can('perm')) return 403`
- 前端隐藏和后端权限不能只依赖其一

---

#### 2. **只读数据仍可写操作 (脱敏可见但权限溢出)**

**快速诊断**
- [ ] 某行数据是脱敏/只读状态，"编辑/删除/转移"按钮是否已隐藏？
- [ ] 批量勾选框是否禁用了无权行的选中？
- [ ] 质检看板/对账单页的按钮是否与用户角色绑定？

**修复要点**
- 按钮需双重检查：`v-if="can('action.key') && isAllowed(item.property)"`
- 列表行的勾选禁用：`:disabled="!canEdit(item)"`
- 字段级权限和操作权限分离

---

#### 3. **查询未过滤租户范围 (跨租户数据泄漏)**

**快速诊断**
- [ ] 所有 SQL SELECT 都含 `WHERE org_id=?` 或 `WHERE property_id IN (...)`？
- [ ] 聚合统计(报表)是否先过滤再计算，不是计算后再过滤？
- [ ] 关系表 JOIN 时，每表都加了租户字段检查？
- [ ] 缓存键是否包含租户维度？如 `cache:case:${orgId}:${caseId}`

**修复要点**
```sql
-- ✗ 不安全
SELECT * FROM cases WHERE status='进行中'

-- ✓ 安全
SELECT * FROM cases 
WHERE status='进行中' 
  AND property_id IN (SELECT id FROM properties WHERE se_id = ?)
```
- 数据范围参数从前端→中间件→DB 全程传递

---

#### 4. **高危操作缺二次确认/受控原因 (误操作无法恢复)**

**快速诊断**
- [ ] 结案/撤案/冲正/作废操作是否都弹了 Modal 确认？
- [ ] Modal 中的"原因"是 `<select>` 下拉还是 `<input>` 自由文本？
- [ ] 后端是否校验原因必须来自 `CFG.REASONS` 枚举？
- [ ] 审计日志是否记录了原因字段？

**修复要点**
- 所有删除/反向操作都需 `Modal.confirm()`
- 原因必须：`<select options={CFG.CLOSE_REASONS}>`
- 前端禁用提交直到选择了原因：`:disabled="!selectedReason"`
- 后端：`validate(reason IN CFG.CLOSE_REASONS)`

---

#### 5. **批量操作无逐项校验 (权限聚合漏洞)**

**快速诊断**
- [ ] 批量派单/标记/删除时，后端是否对每项都检查权限？还是只检查一次？
- [ ] 是否校验了批量操作的"目标对象"(targetVL/targetOrg)归属性？
- [ ] 前端勾选框是否禁用了无权行？

**修复要点**
```js
// ✗ 不安全：仅全局检查一次
if(user.can('case.dispatch')) {
  for(const caseId of caseIds) {
    dispatch(caseId, targetVL);  // 无逐项校验
  }
}

// ✓ 安全：逐项校验四层
for(const caseId of caseIds) {
  const caseData = find(caseId);
  validate(user.can('case.dispatch'));        // 1. 权限
  validate(caseData.property_id IN scope);    // 2. 范围
  validate(targetVL IN user.authorizedVLs);   // 3. 目标
  validate(caseData.status === '待派单');      // 4. 状态
  dispatch(caseId, targetVL);
}
```

---

#### 6. **结案/脱敏规则不完整 (敏感信息泄漏)**

**快速诊断**
- [ ] 结案后 VL/CO 进详情页，是否被重定向到脱敏视图？还是仍加载原页面？
- [ ] 列表行的业主电话/证件号是否在结案后脱敏为 `****`？
- [ ] 子数据(通话记录/承诺/工单)是否都检查了结案状态？

**修复要点**
```js
// ✓ 详情页脱敏重定向
mounted() {
  if (this.case.isClosed && ['VL', 'CO'].includes(this.role)) {
    this.$router.push(`/cases/${this.caseId}/stats`);  // 到脱敏视图
  }
}

// ✓ 列表脱敏
redactField(value, case, field) {
  const shouldRedact = case.isClosed && !['SA', 'SE', 'PL'].includes(this.role);
  if (shouldRedact && REDACT_FIELDS.includes(field)) {
    return '****';
  }
  return value;
}
```
- 脱敏规则集中定义(CFG.REDACT_RULES)，散落各处易失漏

---

#### 7. **用户故事全链路无缺陷 (综合流程验证)**

**快速诊断**
选定一个关键业务流程，以各角色完整走一遍：

**案例：催收员结案流程 (CO视角)**
- [ ] 步骤1：登录 → 工作台待办(应仅见本商案件)
- [ ] 步骤2：进列表 → 筛选进行中 → 进案件(应见通话/标注区)
- [ ] 步骤3：标注 → 填承诺 → 点"结案" → 弹Modal+原因下拉 → 确认结案
- [ ] 步骤4：返回列表 → 案件状态变"已结清" → **再进案件应被重定向脱敏视图**
- [ ] 步骤5：查操作日志 → 仅见本商日志

**修复要点**：任何环节失败即整体失败

### 🟡 MEDIUM优先级（本周内）

#### 8. **代理操作审计痕迹缺失 (操作追溯困难)**

**快速诊断**
- [ ] SA 代 PL 结案时，操作日志是否记录了 `proxyFor` 字段？
- [ ] 被代人查看操作历史时能否识别"谁代自己操作的"？
- [ ] 角色切换后，前端权限缓存是否刷新？(localStorage/sessionStorage 是否清除)

**修复要点**
```js
// ✗ 缺失proxyFor
logOp({
  operator: req.user.id,      // SA-Alice
  action: 'close',
  targetCaseId: req.body.caseId,
  // 缺少代理人信息！
});

// ✓ 完整记录
logOp({
  operator: req.user.id,       // SA-Alice
  action: 'close',
  targetCaseId: req.body.caseId,
  proxyFor: context.proxyFor,  // PL-Bob
  proxyOrgId: context.proxyOrgId,
  reason: req.body.reason,
});
```
- 代理模式下权限范围应收窄(SA 仅操作该 PL 可见的数据)

---

#### 9. **跨端权限口径不同步 (PC/App/H5差异)**

**快速诊断**
- [ ] App 和 PC 的结案脱敏规则是否一致？
- [ ] H5 业主页是否有 token 失效机制？(注销后旧 token 仍可访问吗)
- [ ] PC/App 菜单差异是否文档化或被隐藏？

**修复要点**
- 创建"PC vs App 功能差异表"(若有差异)
- H5 token 应包含过期时间：`token.expiry < now()`
- 脱敏规则在各端应逻辑相同(或集中定义)

---

#### 10. **权限点与代码绑定完整 (无孤岛)**

**快速诊断**
- [ ] 权限字典中的所有点都在代码中被检查过吗？
- [ ] 代码中检查的权限点是否都在字典中定义？
- [ ] 新增功能是否都加了对应的权限点？

**修复要点**
```bash
# 列出权限字典
grep -n "permission:" backend/config/permissions.ts

# 列出所有@Perm使用
grep -rn "@Permission" backend/src/ | sed 's/.*@Permission(\x27\(.*\)\x27).*/\1/' | sort -u

# 对比：是否所有权限字典点都被使用?
# 是否所有can()调用都对应@Permission装饰器?
```

---

#### 11. **API 参数目标校验缺失 (参数竞态)**

**快速诊断**
- [ ] 派单时 targetVL 是否校验了"属于用户授权范围"？
- [ ] 转移案件时 targetProperty 是否校验了所有权？
- [ ] 所有 ID 查询都校验了"关系归属"吗？

**修复要点**
```js
// ✗ 仅校验权限
if(user.can('case.dispatch')) {
  dispatch(caseId, targetVL);  // targetVL 无校验
}

// ✓ 校验参数有效性
if(user.can('case.dispatch')) {
  validate(targetVL IN user.authorizedVLs);  // targetVL 必须属于用户
  validate(caseId IN user.dataRange);        // 目标数据必须在范围内
  dispatch(caseId, targetVL);
}
```

### 🟢 LOW优先级（优化项）

#### 12. **演示数据覆盖多租户 (权限隐形化)**

**快速诊断**
- [ ] Mock 数据是否包含 ≥3 个租户维度(物业/服务商)？
- [ ] SE 的权限过滤在演示中是否能看出差异？还是所有 mock 都属于同一租户？
- [ ] 审计报告是否标注"此缺陷因演示数据弱而不可见"？

**修复要点**
- Mock 数据≥3 个物业，权限过滤才能可视化验证
- 若发现缺陷只在演示中"看不出"，应在报告中标注

## 🚨 常见易错点速查

| 问题 | 原因 | 快速检查 |
|---|---|---|
| **用户看到不该看的数据** | SQL缺WHERE / 脱敏遗漏 / 结案后仍见详情 | grep SELECT \| grep mounted\|isClosed |
| **用户操作了不该操作的** | 后端缺@Permission / 批量无逐项 / 参数无校验 | grep @Permission \| grep for-loop \| grep targetVL |
| **代理操作无痕迹** | 日志缺proxyFor / 权限缓存未清除 | grep logOp \| grep localStorage.clear |
| **跨端不一致** | 脱敏规则差异 / token无过期 | 三端对比 + token检查 |
| **高危操作可误点** | 无Modal确认 / 原因为自由文本 | grep confirmDialog \| grep select |

## 🔧 最小修复集(必须)(2-3天可完成)

1. **所有查询都加 WHERE org_id 或 dataRange 过滤**
2. **所有路由都加 @Permission 装饰器**
3. **所有删除/高危操作都弹二次确认 Modal**
4. **批量操作都用 for-loop 逐项校验权限+范围+目标**
5. **结案后 VL/CO 重定向到脱敏视图(不是仅隐藏标签页)**
6. **操作日志都记录 proxyFor 和 reason 字段**
