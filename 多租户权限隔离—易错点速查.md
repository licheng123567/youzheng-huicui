# 多租户SaaS权限隔离 — 易错点速查表

> 基于有证慧催6轮审计的真实缺陷总结。快速定位问题所属类别与常见原因。

---

## 一、跨租户数据泄漏（最严重）

### 症状
- SE看到不授权物业的案件
- PL-A能查看PL-B的数据
- VL看到其他服务商的信息
- 统计报表包含无权数据

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **SQL WHERE缺租户字段** | DAO/Service SELECT | `grep "SELECT.*WHERE" backend/` 确认每条都有org_id过滤 |
| **前端过滤→后端API无此参数** | 中间件/拦截器 | trace请求：前端传 dataRange 参数→中间件捕获→传给Service |
| **统计聚合无范围限制** | Service 聚合逻辑 | 聚合前先WHERE过滤，不能聚合后再过滤(数据已暴露) |
| **关系表JOIN仅主表过滤** | SQL JOIN语句 | 每表都需 `ON table.org_id = ?` |
| **缓存键未包含租户维度** | Redis key | 应为 `cache:case:${orgId}:${caseId}` 而非 `cache:case:${caseId}` |

### 快速诊断代码片段

```bash
# 检查是否所有SELECT都有范围过滤
grep -n "SELECT.*FROM" backend/src/dao/*.ts | grep -v "WHERE.*org_id"

# 检查是否所有关键API都有数据范围参数
grep -n "async.*req.*res" backend/src/controller/*.ts | head -20
# 逐个验证是否有 const scope = req.user.dataRange 这样的代码
```

## 二、菜单可见但操作不受限（RBAC越权）

### 症状
- 前端隐藏了"成员管理"菜单，但直链 `/settings/members` 可访问
- 按钮隐藏了(v-if="false")，但点击API调用仍成功
- "只读"数据行上"删除"按钮隐藏，但调用DELETE API仍生效

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **后端无权限装饰器** | 路由/Controller | `grep "@Permission\|@Perm" backend/` 确认所有路由都有 |
| **权限装饰器形同虚设** | Controller方法 | `@Perm('case.mark') async mark() {}` 内是否真的校验了？ |
| **前端v-if隐藏，API无二次校验** | 前后分离点 | curl -X POST /api/case/mark 绕过前端检查，应被拒 |
| **导出/下载/打印无权限** | Service导出逻辑 | 这些操作常被遗漏，grep "export\|download\|print" |
| **高危操作缺权限叠加** | Service 删除/转移逻辑 | 删除应检查：权限 + 状态 + 范围，缺一不可 |

### 快速诊断代码片段

```bash
# 检查所有Controller方法是否有权限装饰器
grep -A 1 "async.*(" backend/src/controller/*.ts | grep -B 1 "@Perm" | wc -l

# 检查权限装饰器后的方法内是否真的做了校验
grep -A 5 "@Perm" backend/src/controller/*.ts | grep "can\|Permission\|authorize"

# 检查导出类操作是否有权限
grep -n "export\|download" backend/src/controller/*.ts | grep -v "@Perm"
```

## 三、脱敏规则不完整（结案后仍可见敏感信息）

### 症状
- 结案案件VL仍能进详情页查看完整通话记录
- 列表中结案案件的业主手机号未脱敏
- 结案承诺详情(◐只读显示)而非完全不可见

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **结案后仍加载详情页** | 详情页路由/组件 | 进案件详情时是否先check状态？ if(closed) redirect('/stats') |
| **列表脱敏仅前端** | 渲染函数 | `${maskPhone(phone)}` 是否对所有角色执行？ |
| **子数据(标签页)隐藏而非脱敏** | 标签页渲染 | <TabPane 内 v-if="!isClosed || role==='SA'" 是否覆盖全部子组件？ |
| **通话记录/承诺/工单脱敏不一致** | 各子组件 | 脱敏规则应集中定义(CFG.REDACT_RULES)而非散落各处 |
| **H5业主页结案后仍可见** | H5应用 | H5缺token失效机制，旧token仍可访问 |

### 快速诊断代码片段

```bash
# 检查详情页是否有结案状态检查
grep -n "isClosed\|isClosedCase" frontend/src/pages/case-detail.vue

# 检查脱敏函数是否覆盖所有敏感字段
grep -n "mask\|redact" frontend/src/utils/format.ts | wc -l

# 检查各子组件是否都有脱敏条件
grep -rn "v-if.*isClosed" frontend/src/components/ | wc -l
```

## 四、批量操作权限漏洞（无逐项校验）

### 症状
- 批量派单时后端仅校验一次权限，未逐项验证
- 批量标记风险时target实际不属于caller
- 批量下载对账单对SE无范围过滤

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **for-loop内缺逐项检查** | Service批量方法 | for(id of ids) { validate(id) } 内只有1-2行检查？ |
| **后端仅校验权限不校验范围** | 批量API处理 | 仅 if(user.can('op')) 无 if(item in scope) |
| **前端勾选无权限限制** | 列表组件 | <checkbox :disabled="!canEdit(item)"> 未禁用无权行 |
| **targetVL/targetOrg无授权校验** | 派单逻辑 | 是否validate(target IN user.authorized) ？ |
| **导出/批量下载无范围过滤** | 导出服务 | 应传 dataRange，若为undefined则返回empty |

### 快速诊断代码片段

```bash
# 检查批量操作是否有for-loop逐项校验
grep -A 20 "batchDispatch\|batchUpdate\|batchDelete" backend/src/service/*.ts | grep -E "for|while|forEach" | head -5

# 检查循环内是否有权限检查
grep -A 10 "for.*of.*ids" backend/src/service/*.ts | grep "validate\|can\|Permission" | wc -l

# 前端勾选框是否有禁用条件
grep -n ":disabled=" frontend/src/components/*list*.vue | grep -i check
```

## 五、高危操作缺二次确认（误操作无法恢复）

### 症状
- "作废"直接执行，未弹对话
- 结案时虽弹对话，但原因是自由文本而非下拉
- 冲正回款无"不可逆"提示

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **未弹确认Modal** | 按钮click处理器 | `handleClose() { this.$confirm(...) }` 或 `confirm()` 调用? |
| **原因为自由文本** | Modal弹窗form | `<input type="text">` 而非 `<select options={CFG}>` |
| **未禁用提交按钮** | Modal footer | :disabled="!selectedReason" ？ |
| **后端未校验原因** | Service操作 | validate(reason IN CFG.REASONS) ？ |
| **审计日志无原因记录** | 日志Service | log.reason 是否被持久化? |

### 快速诊断代码片段

```bash
# 检查是否所有高危操作都调用了confirm
grep -n "handleClose\|handleDelete\|handleRevoke" frontend/src/ -r | \
  while read f; do grep -A 3 "$f" | grep -i "confirm\|dialog"; done

# 检查Modal中原因字段是否为select
grep -B 5 -A 5 "原因\|reason" frontend/src/components/*confirm*.vue | grep select

# 检查后端是否校验原因枚举
grep -n "CFG.CLOSE_REASONS\|REASON_ENUM" backend/src/ -r | wc -l
```

## 六、代理操作无审计痕迹（被代人查询时看不出代理关系）

### 症状
- SA代PL结案，操作日志仅显示"SA"，看不出被代人
- 被代PL查看操作历史时无法识别"谁代自己操作的"
- 权限缓存未刷新，代理模式下仍用代理者的范围

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **日志缺proxyFor字段** | logOp()函数 | `logOp(action, target, proxyFor)` 第3参数是否传? |
| **日志保存不完整** | 日志Model | 数据库是否有 operator, proxyFor, targetOrg 三字段? |
| **权限缓存未刷新** | 全局状态 | 切换角色后 localStorage/sessionStorage 是否清除? |
| **代理权限范围未收窄** | 权限查询 | SA代PL时，是否限制SA仅操作该PL可见的数据? |

### 快速诊断代码片段

```bash
# 检查logOp调用是否传proxyFor
grep -n "logOp" backend/src/ -r | head -10
# 对每处调用验证: logOp(action, target, context.proxyFor)

# 检查日志表schema是否有proxyFor字段
sqlite3 app.db ".schema operation_log" | grep proxy

# 检查前端权限缓存刷新逻辑
grep -n "switchRole\|changeRole" frontend/src/ -r -A 5 | grep "localStorage\|sessionStorage\|clearCache"
```

## 八、权限点字典孤岛（定义了但未使用，或用到但未定义）

### 症状
- 权限字典中有`case.evidence`但从未被检查
- 代码中检查`can('case.promise')`但权限字典中无此项
- 权限名大小写不一致导致检查失败

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **权限字典未维护** | permissions.ts或DB | 权限表是否为最新? |
| **前后端权限名不同步** | 前端调用vs后端装饰器 | can('case.mark') vs @Perm('case.update') ？ |
| **新增功能未加权限点** | 新路由/新按钮 | 新增操作是否都有对应权限? |
| **废弃权限未清理** | 权限表 | 是否存在"定义了但无任何代码使用"的权限? |

### 快速诊断代码片段

```bash
# 列出所有权限点定义
grep -n "CFG.PERMISSIONS\|permission:" backend/config/permissions.ts

# 列出所有@Perm使用
grep -rn "@Perm\|@Permission" backend/src/ | sed 's/.*@Perm.*(\x27\(.*\)\x27).*/\1/' | sort -u

# 列出所有can()调用
grep -rn "can(" frontend/src/ | sed 's/.*can(\x27\(.*\)\x27).*/\1/' | sort -u

# 对比：是否所有can()调用都对应@Perm装饰器?
diff <(grep "can(" frontend/ -r | sed 's/.*can(\x27\(.*\)\x27).*/\1/' | sort -u) \
     <(grep "@Perm" backend/ -r | sed "s/.*@Perm.*('\(.*\)').*/\1/" | sort -u)
```

## 十一、结案脱敏规则不完整

### 症状
- 结案后VL/CO详情页仍可见原始数据
- 列表行未脱敏
- 子数据(承诺/工单)仅隐藏未脱敏

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **详情页无状态检查** | 详情路由 | mounted() { if(closed) this.$router.push('/stats') } ？ |
| **列表渲染无脱敏逻辑** | 列表组件 | `<td>{{ redact(row.phone, row) }}</td>` ？ |
| **子组件脱敏不一致** | 各子标签页 | CallRecords/Promises/Cases都检查了closed? |

### 参考修复代码

```js
// ✓ 详情页脱敏重定向
mounted() {
  if (this.case.isClosed && ['VL', 'CO'].includes(this.role)) {
    this.$router.push(`/cases/${this.caseId}/stats`);
  }
}

// ✓ 列表字段脱敏
redactField(value, case, field) {
  const shouldRedact = case.isClosed && !['SA', 'SE', 'PL', 'PC'].includes(this.role);
  if (shouldRedact && REDACT_FIELDS.includes(field)) {
    return '****';
  }
  return value;
}

// ✓ 子组件隐藏
<div v-if="!shouldRedact(case)">
  <call-records :case="case" />
  <promises :case="case" />
</div>
```

## 十二、用户故事走查失败

### 症状
- 整体流程中某环节权限异常
- 流程中权限缓存未更新
- 跨环节脱敏规则不一致

### 常见原因与检查点

| 原因 | 代码位置 | 检查方法 |
|---|---|---|
| **某环节权限校验缺失** | 多处 | 逐步 walk through，每步记录权限检查是否生效 |
| **权限缓存生命周期混乱** | 全局状态管理 | 页面跳转时是否清除缓存? 角色切换时是否刷新? |
| **脱敏规则在环节间不一致** | 多个组件 | 同一案件在列表/详情/日志的脱敏规则是否相同? |

### 诊断方法

```
以CO结案为例：
1. 登录→TODO列表（应见本商案件）
   检查：列表数据是否仅本商? SQL WHERE是否含商户ID?
   
2. 进案件详情（应见通话/承诺）
   检查：组件是否加载了正确的数据? dataRange参数是否传递?
   
3. 结案（应弹二次确认）
   检查：Modal是否出现? 原因是否为select?
   
4. 返回列表（案件状态变更）
   检查：列表是否自动刷新? 状态字段是否更新?
   
5. 再进案件（应重定向脱敏视图）
   ← 最容易在这里失败！检查：
     - 路由mounted是否检查状态?
     - 是否真的 this.$router.push('/stats') ?
     - 能否直链绕过? /case/123 是否被拒?
   
6. 查操作日志（应仅见本商）
   检查：日志WHERE是否含org_id过滤?
```

## 速查导航树

```
问题：用户看到不该看的数据
  ├─ 是物业/商户数据? → 检查 #一
  ├─ 是敏感字段? → 检查 #三
  └─ 是结案后信息? → 检查 #十一

问题：用户操作了不该操作的
  ├─ 直链或API调用? → 检查 #二
  ├─ 批量操作? → 检查 #四
  └─ 高危操作? → 检查 #五

问题：审计/追溯困难
  ├─ 代理操作痕迹缺失? → 检查 #六
  └─ 操作原因难以分类? → 检查 #五

问题：跨端不一致
  → 检查 #七

问题：新增功能权限不对
  ├─ 没加@Perm装饰器? → 检查 #八
  └─ 参数未校验? → 检查 #十

问题：整体流程异常
  → 检查 #十二
```

## 黄金诊断路径（3分钟快速定位）

1. **数据泄漏?** → grep SQL WHERE子句 (#一)
2. **权限越权?** → grep @Perm装饰器 (#二)
3. **脱敏失败?** → grep isClosed判断 (#三)
4. **代理无痕?** → grep logOp proxyFor (#六)
5. **整体异常?** → 跑一遍用户故事 (#十二)

## 快速修复清单

| 缺陷 | 修复代码行 | 修复时间 |
|---|---|---|
| SQL缺WHERE | 加 WHERE org_id = #{orgId} | 5min |
| 缺@Perm | 加 @Permission('perm.key') | 5min |
| 脱敏缺失 | 加 if(isClosed) return redactView | 10min |
| 二次确认缺失 | 加 Modal.confirm() | 10min |
| 批量无逐项检查 | 加 for(item of items) validate | 15min |
| 代理日志缺proxyFor | 加 proxyFor 参数到 logOp | 5min |
| 权限点孤岛 | 对标权限字典补充缺失点 | 20min |
