## 🛠️ 五分钟快速诊断

### 症状：用户看到不该看的数据

```
是多租户数据泄漏？
  ├─ 查SQL: grep "SELECT.*WHERE" | grep -v org_id
  └─ 修复: 加 WHERE org_id = #{orgId}

是敏感字段未脱敏？
  ├─ 查代码: grep "isClosed.*redact"
  └─ 修复: 加脱敏条件 if(isClosed) return mask()

是结案后仍见详情？
  ├─ 查路由: if(isClosed) this.$router.push('/stats')
  └─ 修复: 加重定向逻辑
```

### 症状：用户操作了不该操作的

```
是直链或API调用越权？
  ├─ 查后端: grep "@Perm" 确认装饰器
  └─ 修复: 加 @Permission('perm.key') 装饰器

是批量操作无逐项检查？
  ├─ 查代码: grep -A 20 "batch" | grep "for\|while"
  └─ 修复: for(item of items) { validate(item) }

是高危操作缺确认？
  ├─ 查前端: 点击操作，是否弹Modal
  └─ 修复: 加 Modal.confirm() 和原因下拉
```

### 症状：审计追溯困难

```
是代理操作无痕迹？
  ├─ 查日志: 是否有 proxyFor 字段
  └─ 修复: logOp(action, target, context.proxyFor)

是操作原因难分类？
  ├─ 查表单: 原因是 <select> 还是 <input>
  └─ 修复: 用 <select options={CFG.REASONS}>
```

## 🎓 典型缺陷案例集

### 缺陷1：RBAC越权 — 前端隐藏，后端无校验

**场景**：有证慧催 v06 中"成员管理"功能

```
前端: <menu v-if="role === 'SA'">成员管理</menu>  ✓ 隐藏正确

后端: @GetMapping("/api/members")  ✗ 无权限检查
      async getMembers() {
        return memberService.findAll();  // 直接返回全部
      }

后果：非SA用户直链 /api/members 仍可获取全部成员列表

修复：
@GetMapping("/api/members")
@Permission("member.view")  ← 加权限装饰器
async getMembers() {
  validate(user.can('member.view'));  // 二次校验
  return memberService.findAll();
}
```

### 缺陷2：数据泄漏 — SE查询无范围过滤

**场景**：有证慧催 v08 中"质检看板"

```
SQL(不安全):
SELECT * FROM qc_alerts WHERE status = 'high_risk'
// 返回全平台高风险告警

SE-Alice看到: 所有物业的告警(应仅见授权范围)

修复后:
SELECT * FROM qc_alerts 
WHERE status = 'high_risk' 
  AND property_id IN (
    SELECT property_id FROM se_datarange 
    WHERE se_id = #{currentUserId}
  )
```

### 缺陷3：脱敏失败 — 结案后仍进详情页

**场景**：有证慧催 v06 报告中"CO访问结案案件"

```
前端(不安全):
template:
  <div v-if="!case.isClosed">
    <tab-pane label="通话记录">...</tab-pane>
    <tab-pane label="承诺">...</tab-pane>
  </div>
  <div v-else>
    <div>已结清</div>  ← 仍在详情页，仅隐藏标签
  </div>

问题：VL/CO在结案后仍可访问详情页，虽然标签页隐藏了，
     但页面结构暴露了"这个案件曾经有通话/承诺"

修复后:
mounted() {
  if (this.case.isClosed && ['VL', 'CO'].includes(this.role)) {
    this.$router.push(`/cases/${this.caseId}/stats`);  ← 重定向
  }
}

// 脱敏统计视图：仅显示计数/转化率，无原始数据
<stats-summary :case="case" />  ← 专用脱敏组件
```

### 缺陷4：批量无逐项校验 — 派单全部通过

**场景**：有证慧催 v07 中"批量派单"

```
不安全代码:
async batchDispatch(req) {
  if (!user.can('case.dispatch')) {  ← 仅检查一次
    throw new Error('Unauthorized');
  }
  
  for (const caseId of req.body.caseIds) {
    await dispatch(caseId, req.body.targetVL);  ← 无逐项校验
  }
}

攻击场景：
POST /api/case/batch-dispatch {
  "caseIds": [1, 2, 3],
  "targetVL": "unauthorized-vendor-xyz"  ← 未授权的服务商
}
→ 派单仍成功(危险!)

修复后:
async batchDispatch(req) {
  const { caseIds, targetVL } = req.body;
  
  for (const caseId of caseIds) {  ← for-loop逐项校验
    const caseData = await caseService.find(caseId);
    
    // 校验1：权限
    validate(user.can('case.dispatch'));
    
    // 校验2：数据范围
    validate(caseData.property_id IN user.dataRange);
    
    // 校验3：目标对象
    validate(targetVL IN user.authorizedVLs);
    
    // 校验4：状态
    validate(caseData.status === '待派单');
    
    await dispatch(caseId, targetVL);
  }
}
```

### 缺陷5：代理无痕迹 — 操作日志缺proxyFor

**场景**：有证慧催 v08 报告[A项]"SA代PL结案未记代理人"

```
不安全代码:
async doClose(req) {
  const result = await caseService.close(req.body);
  
  logOp({
    operator: req.user.id,  // SA-Alice
    action: 'close',
    targetCaseId: req.body.caseId,
    // ✗ 缺少代理人信息！
  });
  
  return result;
}

问题：PL-Bob查操作日志，看到"SA-Alice结案了我的案件"
     但无法追溯为什么SA要代自己操作

修复后:
async doClose(req, context) {
  const result = await caseService.close(req.body);
  
  logOp({
    operator: req.user.id,     // SA-Alice
    action: 'close',
    targetCaseId: req.body.caseId,
    proxyFor: context.proxyFor,      // ← PL-Bob
    proxyOrgId: context.proxyOrgId,  // ← 物业A
    reason: req.body.reason,
  });
  
  return result;
}
```

```
┌─────────────────────────────────────┐
│ 防线1：导航与权限一致                │
│ 前端菜单隐藏 + 后端权限校验          │

└─────────────────────────────────────┘
          ↓
┌─────────────────────────────────────┐

│ 只读字段无写按钮 + 租户过滤 + 脱敏  │
│ 检查项: #2, #3, #11                 │
└─────────────────────────────────────┘
          ↓
┌─────────────────────────────────────┐
│ 防线3：高危操作的受控流程            │
│ 二次确认 + 操作追溯 + 逐项校验       │
│ 检查项: #4, #5, #6                  │
└─────────────────────────────────────┘
```

### 优先级判定规则

| 缺陷数 | 评级 | 处理 |
|---|---|---|
| 🔴 HIGH全通过 | 一/二级 | 可交付 |
| 🔴 HIGH ≤2失 | 二级 | 可交付 + 修复承诺 |
| 🔴 HIGH >2失 | 三级 | 禁交 + 必须修复 |

### 最小修复集(必须)

1. **SQL所有表都加WHERE org_id**
2. **所有路由都加@Permission装饰器**
3. **所有高危操作都弹二次确认**
4. **批量操作都for-loop逐项校验**
5. **结案后VL/CO重定向到脱敏视图**

## 📞 常见问题

**Q: 这个体系适用于哪些项目？**
A: 所有多租户SaaS系统。核心12项通用，具体检查命令可按技术栈调整。

**Q: 全部审计需要多长时间？**
A: 代码审查30min/PR；功能测试2h/版本；上线前4h；月度6h。

**Q: 如果发现严重缺陷应该怎么办？**
A: 按《快速修复工作表》对症下药，通常1-2天内可修复。

**Q: 权限矩阵和这个体系什么关系？**
A: 权限矩阵是"规范"(定义应该怎样)，审计体系是"验证"(确认实际是否如此)。
